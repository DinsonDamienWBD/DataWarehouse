## STEP 0: v7.0 GAP CLOSURE (1 Phase)

### Phase 111.5: Close Deferred v7.0 Requirements
**Gaps:** MUTN-02, SOAK-01, SOAK-04 — three requirements left unchecked when v7.0 shipped.

**Current State:**
- **MUTN-02** (Surviving mutants documented): Phase 104 was marked N/A because the hardening test suite only does source-code analysis (reads .cs files, validates patterns via regex). Stryker mutates IL code, so mutations are invisible to these tests. No meaningful Stryker run was ever completed. Needs a runtime test project that calls production methods directly.
- **SOAK-01** (Full 52-plugin kernel boot + 100GB VDE streaming): Kernel bootstrap harness built, all 5 kernel tests pass (boots 52 plugins via reflection). But the 100GB VDE streaming test is Skip-annotated — never executed. Only 10MB sanity test ran.
- **SOAK-04** (Gen2 collection rate + working set monotonic growth): Only a 2-minute soak completed. Gen2 rate 1.00/min (PASS), but working set grew 27.62% (113→144MB). Attributed to warm-up but not validated with longer run. Recommendation was 10-minute and 24-hour runs.

**Implementation Plan (5 plans):**
1. **111.5-01**: Runtime unit test project — Create `DataWarehouse.Runtime.Tests` with tests that call production SDK methods directly (not source-code analysis). Target key code paths: BoundedCache operations, VDE decorator chain methods, policy engine resolution, message bus publish/subscribe, strategy base contract methods. Must exercise IL-level code so Stryker mutations are visible.
2. **111.5-02**: Stryker mutation run + survivor documentation — Run Stryker on `DataWarehouse.Runtime.Tests` against SDK and core projects. Target 95%+ mutation score. Document all surviving mutants with justification (equivalent mutants, unreachable code, acceptable risk). Output: `Metadata/stryker-report/surviving-mutants.md`. Satisfies **MUTN-02**.
3. **111.5-03**: Execute VDE streaming soak test — Un-skip the 1GB VDE streaming test (100GB impractical in CI). Run full VDE decorator chain write + readback + checksum verification. If 1GB passes, document 100GB as manual-only certification step. Satisfies **SOAK-01**.
4. **111.5-04**: Extended soak test (10-minute) — Run the existing soak test harness with `SOAK_DURATION_MINUTES=10` to amortize .NET warm-up. Verify: Gen2 rate stays <2/min, working set plateaus after warm-up (no monotonic growth trend in final 5 minutes). Document results. Satisfies **SOAK-04**.
5. **111.5-05**: SharpFuzz VDE corruption harness — Comprehensive binary fuzzing harness using SharpFuzz targeting VDE block allocation, compression, format parsing, integrity, and journal recovery paths under corrupted input. Targets:
   - **Block allocation** (`VirtualDiskEngine/BlockAllocation/`):
     - `Metaslab.EnsureBitmapLoadedAsync()` — bitmap deserialization from device blocks, `RebuildFreeLists()` building free-run lists from corrupt bitmap bytes
     - `MetaslabAllocator` — morph-level selection and delegation to `BitmapAllocator`/`AllocationGroupDescriptorTable`/`MetaslabDirectory`
     - `BitmapAllocator` — flat bitmap scan with `GetBitmapSnapshot()` feeding `ExtentTree.BuildFromBitmap()`
     - `FreeSpaceManager` — coordinated bitmap + extent tree allocation under corrupt state
     - `ExtentTree` + `PersistentExtentTree` — extent serialization/deserialization with XxHash64 checksum validation, `LoadAsync()` parsing block chains with `[MagicEXTR:4][Version:2][ExtentCount:4][Extents*N][XxHash64:8]` format
     - `SimdBitmapScanner` — SIMD-accelerated free-block scan with corrupt bitmap input
     - Feed: corrupted serialized allocation bitmaps, AG descriptors with invalid offsets/counts, extent records with overlapping ranges or out-of-bounds block addresses, corrupt XxHash64 checksums in extent blocks
   - **Allocation groups** (`VirtualDiskEngine/Allocation/`):
     - `AllocationGroup` + `AllocationGroupDescriptorTable` — AG descriptor parsing, group-level bitmap
     - `SubBlockPacker` + `SubBlockBitmap` — sub-block packing with corrupt sub-block metadata
     - `ExtentTree` + `ExtentTreeNode` — extent tree node traversal with corrupt child pointers
   - **Compression** (`VirtualDiskEngine/Compression/` + `Pipeline/Stages/`):
     - `PerExtentCompressor.DecompressExtentAsync()` — Brotli decompression from `CompressedExtent` with corrupt `OriginalSize`/`CompressedSize`/`Data` fields
     - `DataTransformerStage` — pipeline compression/decompression stage processing corrupt `VdePipelineContext.DataBuffer`
     - Feed: truncated compressed blocks, blocks with valid Brotli headers but corrupted payloads, `CompressedExtent` claiming `OriginalSize` >> actual decompressed output, mismatched compression algorithm IDs, zero-length data with non-zero `OriginalSize`
   - **Format parsing** (`VirtualDiskEngine/Format/` + `Container/` + `Metadata/`):
     - `Superblock.Deserialize()` — v1 superblock from raw bytes with corrupt magic, invalid `blockSize` (non-power-of-2, <MinBlockSize, >MaxBlockSize), negative `TotalBlocks`/`FreeBlocks`, mismatched XxHash64
     - `SuperblockV2.Deserialize()` — v2 superblock with corrupt `ModuleManifest`, invalid `InodeSize`, corrupt `VersionInfo`
     - `SuperblockGroup.DeserializeFromBlocks()` — multi-block superblock group parsing
     - `MagicSignature.Deserialize()` — magic signature validation from corrupt first 16 bytes
     - `CompactInode64` (64-byte), `InodeV2`/`Inode.Deserialize()` (256-byte), `ExtendedInode512` (512-byte) — inode parsing with corrupt type enums, negative sizes, invalid block pointers, corrupt symlink target lengths
     - `MixedInodeAllocator.DetectFormat()` — format detection from ambiguous buffer sizes
     - `InodeExtent` — extent records with out-of-bounds `StartBlock`, negative `BlockCount`, sparse flag manipulation
     - `RegionDirectory.Deserialize()` + `RegionPointerTable.Deserialize()` — region metadata parsing with corrupt slot entries
     - `ModuleManifest` + `ModuleConfig` — module bit parsing with invalid module IDs
     - `UniversalBlockTrailer.Verify()` — trailer checksum verification with corrupt payloads
     - `VdeDetector.Detect()` + `VdeFormatDetector.DetectFromBytes()` — format detection from adversarial byte sequences
     - `PreambleHeader.Deserialize()` — preamble header parsing with corrupt VDE offset
   - **Integrity** (`VirtualDiskEngine/Integrity/`):
     - `HierarchicalChecksumTree.VerifyBlockChecksum()` — block XxHash64 verification with corrupt trailers
     - `CrossExtentIntegrityChain.VerifyBlocksAsync()` — cross-extent TRLR verification with corrupt per-block records
     - `EpochBatchedMerkleUpdater` — Merkle tree updates with corrupt epoch data
     - `BlockChecksummer` + `ChecksumTable` — checksum table parsing with corrupt entries
   - **Journal/WAL** (`VirtualDiskEngine/Journal/`):
     - `JournalEntry` — journal entry deserialization with corrupt `SequenceNumber`, `TransactionId`, `BeforeImage`/`AfterImage`
     - `WriteAheadLog` + `ShardedWriteAheadLog` — WAL replay from corrupt log blocks
     - `CheckpointManager` — checkpoint recovery from corrupt checkpoint data
     - `WalTransaction` — transaction boundaries with interleaved corrupt entries
   - **CoW/Dedup** (`VirtualDiskEngine/CopyOnWrite/`):
     - `ExtentAwareCowManager.DeserializeRefCounts()` — ref count table parsing (already has corrupt-input guard — verify it holds under fuzz)
     - `ContentAddressableDedup` — dedup hash table with corrupt content-address entries
     - `SnapshotManager` — snapshot metadata parsing with corrupt inode references
   - **Assertions**: No unhandled exceptions escape (all corruption caught and wrapped in `VdeCorruptionException`, `InvalidDataException`, or similar). No buffer overruns (verify all `Span<byte>.Slice()` calls are bounds-checked). No infinite loops from circular extent pointers or WAL chains. No OOM from decompression bombs or inflated `OriginalSize`. Working set stays bounded (<500MB) regardless of input. All `ArrayPool` rentals are returned even on corrupt-input paths.
   - **Output**: `DataWarehouse.Fuzz.Tests/` project with SharpFuzz `[FuzzTest]` methods, initial corpus seeded from valid VDE containers created via `VdeCreator.CreateVdeAsync()`, crash reproduction tests for any findings, coverage report showing fuzz-tested code paths. Satisfies **FUZZ-01**.
   - **Skill**: Run `/pentest-fuzz` to analyze any SharpFuzz crash logs and generate patches. **Prerequisite**: The `DataWarehouse.Fuzz.Tests/` project and `[FuzzTest]` harnesses must be built and executed first (this plan) before `/pentest-fuzz` can analyze crash output.

**Effort:** Medium-Large (5 plans, 2 waves — plans 1-2 sequential, plans 3-5 parallel with each other and independent of 1-2)
**Dependencies:** None (all infrastructure exists from v7.0 Phases 104-106)
**Exit Criteria:** All requirements checked: MUTN-02 ✓, SOAK-01 ✓, SOAK-04 ✓, FUZZ-01 ✓

#### Phase 112: SPDK Full Implementation
**Gap:** A.1 — SPDK is defined as `BlockDeviceImplementation.Spdk = 6` in `BlockDeviceOptions.cs` and documented in the VDE spec (AD-41: ~0.3-0.8μs latency), but the actual `SpdkBlockDevice` class is a planned stub. Plan 87-59 was written but never executed.

**Current State:**

- `BareMetalOptimizer.cs` has `ConfigureSpdkOptimizations()` with logging-only implementation
- `UltimateFilesystemPlugin` has `_kernelBypassEnabled` flag and detection
- `DeploymentEnvironment.BareMetalSpdk` enum value exists

**Implementation Plan (8 plans):**
1. **112-01**: `SpdkNativeBindings` — P/Invoke layer to libspdk: `spdk_nvme_ns_cmd_read`, `spdk_nvme_ns_cmd_write`, `spdk_nvme_qpair_process_completions`, `spdk_dma_malloc`, `spdk_dma_free`, `spdk_nvme_ctrlr_alloc_io_qpair`. Conditional compilation for Linux-only (`#if LINUX`). NuGet package `DataWarehouse.Native.Spdk` wrapping prebuilt `libspdk.so`.
2. **112-02**: `SpdkDmaAllocator` — `IMemoryAllocator` implementation using `spdk_dma_malloc/free` for 4KiB-aligned DMA buffers. `SpdkDmaBufferPool` for buffer reuse (arena pattern). `IDisposable` with leak tracking.
3. **112-03**: `SpdkBlockDevice` — `IBlockDevice` + `IBatchBlockDevice` implementation. Constructor takes NVMe controller path + namespace ID. `vfio-pci` handoff sequence. Queue pair per-thread for lock-free I/O. Zero-copy path: SPDK DMA buffer → caller's aligned buffer.
4. **112-04**: `SpdkBlockDeviceFactory` integration — Wire into `BlockDeviceFactory` cascade as highest priority. Auto-detect: check `/sys/bus/pci/drivers/vfio-pci` for NVMe controllers. Graceful fallback when SPDK unavailable.
5. **112-05**: `NvmeNamespaceProbe` — Query NVMe controller via SPDK Identify commands: ZNS support, optimal sector size, max queue depth, NVMe-oF discovery. Feed into `BlockDeviceOptions`.
6. **112-06**: ZNS integration — Map SPDK zone management commands to VDE epoch-zone symbiosis (AD invariant 17). `ZnsSpdkBlockDevice extends SpdkBlockDevice` with `ZONE_RESET`, `ZONE_OPEN`, `ZONE_CLOSE` support.
7. **112-07**: SPDK + CompoundBlockDevice integration — Test SPDK as backing device for RAID 0/1/5/6/10. Verify SPDK DMA alignment through RAID parity calculations. Benchmark: target 10 GB/s sequential, 2M+ IOPS random 4K.
8. **112-08**: SPDK E2E test — Full VDE decorator chain (Cache→Integrity→Compression→Dedup→Encryption→WAL→RAID→**SPDK**) end-to-end. Requires QEMU NVMe emulation for CI or real NVMe hardware.

#### Phase 114: S3 Epoch-Flush Block Device
**Gap:** B.1 — S3EpochFlushBlockDevice for cloud-native VDE storage.

**Current State:**
- S3-compatible server exists in `UltimateInterface` (S3 GET/PUT/DELETE/HEAD/LIST)
- `CrossCloudBackupStrategy` has AWS/Azure/GCP client patterns
- VDE WAL infrastructure fully built (dual-ring, epoch-batched)
- No `IBlockDevice` implementation backed by object storage

**Implementation Plan (5 plans):**
1. **114-01**: `S3EpochFlushBlockDevice` core — `IBlockDevice` over local WAL + S3. Write path: append to local WAL ring → background epoch flush thread packages WAL entries into S3 objects (one object per epoch). Read path: ARC cache L1 → local WAL ring → S3 GET.
2. **114-02**: Epoch object format — Each S3 object = `epoch-{vde_uuid}-{epoch_number}.dwvd-wal`. Contains block writes for that epoch. Manifest object tracks epoch sequence. S3 versioning for atomic epoch commits.
3. **114-03**: Multi-provider abstraction — `IObjectStorageClient` interface with `AwsS3Client`, `AzureBlobClient`, `GcsClient` implementations using official SDKs (AWSSDK.S3, Azure.Storage.Blobs, Google.Cloud.Storage.V1).
4. **114-04**: Recovery and consistency — Mount: download manifest → identify latest consistent epoch → replay from S3 objects. Crash recovery: local WAL has uncommitted epochs → flush on remount or discard.
5. **114-05**: Cost-optimization — Intelligent tiering: hot epochs in S3 Standard, cold epochs auto-tiered to S3 Glacier/Archive. Read-ahead prefetch for sequential scans. Configurable: `S3FlushIntervalMs`, `LocalWalSizeMb`, `TierAfterDays`.

#### Phase 115: WAL Group Commit Optimization
**Gap:** A.9 — Each write goes through full pipeline individually. High-frequency trading needs batched WAL commits.

**Current State:**
- `WalBlockWriterStage.cs` writes WAL entry + device block + flush per write
- `ADAPTIVE-SCALING-ANALYSIS.md` documents L3 group commit: "Accumulate 10ms of WAL records, single fsync. Reduces fsync overhead from 1/write to 1/10ms"
- `WalMessageQueue` in scaling infrastructure has batch flush pattern
- `EpochBatchedMerkleUpdater` demonstrates the epoch-batch pattern

**Implementation Plan (3 plans):**
1. **115-01**: `GroupCommitAccumulator` — Collects WAL entries for configurable interval (default 10ms). Uses `Channel<WalEntry>` bounded producer-consumer. Single `fsync` per group. Configurable: `GroupCommitIntervalMs`, `MaxGroupSize`, `FsyncPolicy` (per-op / group-commit / lazy).
2. **115-02**: Pipeline integration — Wire `GroupCommitAccumulator` into `WalBlockWriterStage`. `await` returns only after group fsync completes (durability guarantee maintained). Callers see latency = max(their-arrival-to-group-close, fsync-time). Benchmark: target 10x throughput for small writes.
3. **115-03**: Adaptive group sizing — Under low load, flush immediately (latency priority). Under high load, accumulate larger groups (throughput priority). `BackpressureAware` integration: when WAL backpressure detected, increase group size dynamically.

**Effort:** Small-Medium (3 plans, 1 wave)
