Create `ShardMergeEngine` sealed class. `[SdkCompatibility("6.0.0", Notes = "Phase 93: VDE 2.0B Shard Lifecycle (VSHL-03)")]`.

Constructor takes:
- `Func<Guid, Guid, CancellationToken, ValueTask<long>> dataMigrator` -- migrates all data from source shard VDE to target shard VDE. Returns blocks migrated.
- `Func<Guid, CancellationToken, ValueTask> shardDecommissioner` -- decommissions a VDE shard (cleanup, free resources).
- All parameters non-null (ArgumentNullException.ThrowIfNull).

**Core method:** `async ValueTask<ShardMergeResult> MergeAsync(DataShardDescriptor lowerShard, DataShardDescriptor upperShard, IndexShardCatalog indexCatalog, RoutingTable routingTable, CancellationToken ct = default)`:

Algorithm:
1. **Validate adjacency**: `lowerShard.EndSlot == upperShard.StartSlot`. If not, return `ShardMergeResult.Skipped(lowerShard, upperShard, $"Shards are not adjacent: lower ends at {lowerShard.EndSlot}, upper starts at {upperShard.StartSlot}")`.
2. **Validate order**: `lowerShard.StartSlot < upperShard.StartSlot`. If not, return Skipped with "Shards are not in correct order".
3. **Validate both available**: both `lowerShard.IsAvailable` and `upperShard.IsAvailable`. If not, return Skipped with "One or both shards unavailable".
4. Start a Stopwatch.
5. **Decide survivor**: the lower shard survives (keeps its VDE ID). The upper shard is absorbed and decommissioned. This minimizes data movement when the lower shard typically has more data.
6. **Build merged descriptor**:
   ```
   var mergedShard = new DataShardDescriptor {
       DataVdeId = lowerShard.DataVdeId,  // survivor
       StartSlot = lowerShard.StartSlot,
       EndSlot = upperShard.EndSlot,  // expanded range
       CapacityBlocks = lowerShard.CapacityBlocks + upperShard.CapacityBlocks,
       UsedBlocks = lowerShard.UsedBlocks,  // updated after migration
       Health = DataShardHealth.Healthy,
       LastHeartbeatUtcTicks = DateTime.UtcNow.Ticks
   };
   ```
7. **Migrate data**: `long blocksMigrated = await dataMigrator(upperShard.DataVdeId, lowerShard.DataVdeId, ct)`. Update `mergedShard = mergedShard with { UsedBlocks = lowerShard.UsedBlocks + blocksMigrated }`.
8. **Atomic index update** (within try/catch for rollback):
   a. `indexCatalog.RemoveDataShard(upperShard.DataVdeId)` -- remove absorbed shard
   b. `indexCatalog.RemoveDataShard(lowerShard.DataVdeId)` -- remove old lower
   c. `indexCatalog.AddDataShard(mergedShard)` -- add merged
   If any step throws, attempt rollback: remove merged, re-add both originals. Return Failed result.
9. **Routing table update**: `routingTable.AssignRange(mergedShard.StartSlot, mergedShard.EndSlot, mergedShard.DataVdeId)`.
10. **Decommission absorbed shard**: `await shardDecommissioner(upperShard.DataVdeId, ct)`.
11. Stop stopwatch. Return `ShardMergeResult` with Status=Completed, both original shards, merged shard, DecommissionedVdeId=upperShard.DataVdeId, blocksMigrated, elapsed.

Wrap entire operation in try/catch. On any exception after step 8, return Failed with exception message.

**Batch method:** `async ValueTask<IReadOnlyList<ShardMergeResult>> MergeCandidatesAsync(IReadOnlyList<DataShardDescriptor> candidates, IndexShardCatalog indexCatalog, RoutingTable routingTable, CancellationToken ct = default)`:
1. Candidates are expected pre-sorted by StartSlot (from ShardCapacityMonitor.FindMergeCandidatesAsync).
2. Walk the sorted list looking for adjacent pairs: if `candidates[i].EndSlot == candidates[i+1].StartSlot`, they are merge-eligible.
3. Merge each eligible pair sequentially (not in parallel -- index updates must be serialized).
4. After merging a pair, skip the next candidate (it was absorbed). Continue scanning.
5. Return all merge results.

Thread safety: same as ShardSplitEngine -- callers must serialize merges per shard.
