  <action>
1. Create `Metadata/formal-verification/` directory.

2. **WalConsistency.tla** — Model the WAL dual-ring architecture (Ring A crash recovery + Ring B replication):
   - Variables: `walLog` (sequence of entries), `committedTxns` (set), `diskState` (function from block -> value), `crashPoint` (model value)
   - Actions: `BeginTxn`, `AppendWalEntry`, `CommitTxn`, `FlushToDisk`, `Crash`, `Recover`
   - Safety invariant `NoPartialCommits`: After `Recover`, for every txn T, either ALL of T's writes are in `diskState` or NONE are
   - Safety invariant `WalDurability`: A committed txn's entries survive any single crash
   - Liveness: `CommittedTxnEventuallyFlushed`: Every committed txn eventually reaches disk
   - Model Ring A (crash): entries written to WAL before data pages
   - Model Ring B (replication): entries replicated to N peers before commit ack

3. **RaidParity.tla** — Model RAID parity for the VDE RAID decorator:
   - Variables: `drives` (array of drive states), `parityDrive`, `dataBlocks` (function drive x stripe -> data)
   - Actions: `WriteStripe`, `ReadStripe`, `DriveFail`, `Rebuild`
   - Safety invariant `ParityCorrect`: For every stripe, XOR of all data blocks equals parity block
   - Safety invariant `SingleFailureRecoverable`: After any single `DriveFail`, all data recoverable via `Rebuild`
   - Model both RAID-5 (distributed parity) and RAID-6 (dual parity) configurations

4. **README.md** — Document:
   - Why TLA+ (not Coq/Lean): C# has no proof assistant, TLA+ proves design, property tests prove implementation
   - How to check: install TLA+ tools, run `tlc WalConsistency.tla` etc.
   - Mapping: each TLA+ invariant -> corresponding C# class and property-based test (Phase 132-02)
   - References: AWS use of TLA+, Lamport's TLA+ resources

1. **RaftSafety.tla** — Model Raft consensus as implemented in DataWarehouse:
   - Variables: `currentTerm` (per node), `votedFor` (per node), `log` (per node), `commitIndex` (per node), `state` (Follower/Candidate/Leader per node)
   - Actions: `RequestVote`, `AppendEntries`, `ElectionTimeout`, `ClientRequest`, `Crash`, `Restart`
   - Safety invariant `ElectionSafety`: At most one leader per term
   - Safety invariant `LogMatching`: If two logs have an entry with same index and term, all preceding entries match
   - Safety invariant `LeaderCompleteness`: If entry committed in term T, present in all leaders of terms > T
   - Safety invariant `StateMachineSafety`: No two nodes apply different entries at same log index
   - Based on Ongaro's Raft TLA+ spec but adapted to DataWarehouse's implementation details (e.g., pre-vote extension)

2. **FederationCatalog.tla** — Model the federation shard catalog:
   - Variables: `shardCatalog` (per region), `shardLocations` (function shard -> set of regions), `pendingMigrations`
   - Actions: `CreateShard`, `MigrateShard`, `SplitShard`, `MergeShard`, `PropagateUpdate`, `RegionPartition`, `RegionRecover`
   - Safety invariant `ShardOwnership`: Every shard has exactly one primary region at any time
   - Safety invariant `CatalogConvergence`: After partition heals, all region catalogs converge to same state (CRDT merge)
   - Safety invariant `NoDataLoss`: A shard present before migration is present after migration completes
   - Model CRDT-based catalog updates with last-writer-wins for conflict resolution
  </action>
