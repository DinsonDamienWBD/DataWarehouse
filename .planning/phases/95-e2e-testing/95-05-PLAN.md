The VDE requires a file-based container (VdeOptions.ContainerPath). For E2E tests that need a real VDE instance, use `Path.GetTempFileName()` with `.dwvd` extension and clean up in a finally block. The VDE InitializeAsync creates the container file. This is the legitimate way to test VDE without OS-level devices.

Implement these 7 tests:

1. **TestRawDeviceToPoolBootstrap**: Create 4 InMemoryPhysicalBlockDevice instances (4096 block size, 10_000 blocks each). Create a CompoundBlockDevice with RAID 5 layout over them. Verify BlockSize = 4096. Verify BlockCount = 3 * 10_000 (3 data devices in RAID 5 with 4 devices). Write a test pattern to first 100 logical blocks. Read back and verify byte-exact match. Create a DevicePoolDescriptor with PoolName="test-pool", Tier=Hot, matching member descriptors. Verify descriptor has 4 members, correct capacity. This proves raw-device-to-pool path works.

2. **TestPoolToVdeCreation**: Create a 4-device RAID 5 CompoundBlockDevice as the pool. Create a temp file path for VDE container. Create VdeOptions with ContainerPath=tempFile, BlockSize=4096, TotalBlocks=1024 (small for test speed). Create VirtualDiskEngine, call InitializeAsync. Store a test object via `StoreAsync("test/key1", testData)`. Retrieve via `RetrieveAsync("test/key1")`. Verify retrieved data matches stored data byte-for-byte. Dispose VDE. Delete temp file in finally block. This proves VDE creation on top of pool infrastructure.

3. **TestVdeStoreRetrieveDeleteLifecycle**: Create VDE on temp file (NOT on CompoundBlockDevice — VDE uses file path). Store 10 objects with keys "obj/0" through "obj/9", each 1KB of deterministic data. List all objects — verify 10 returned. Delete "obj/5". List again — verify 9 returned and "obj/5" absent. Retrieve remaining 9 — verify data integrity. This proves the complete CRUD lifecycle within a single VDE.

4. **TestFederationOverMultipleVdes**: Create 2 separate VDE instances on separate temp files (simulating 2 shards). Store 5 objects in VDE-A (keys "shard-a/obj/0" through "shard-a/obj/4") and 5 in VDE-B (keys "shard-b/obj/0" through "shard-b/obj/4"). Create a FederatedVirtualDiskEngine in single-VDE mode with VDE-A first. Verify single-VDE operations work (store/retrieve). Then verify that VDE-B also operates independently. This tests the federation wrapper's single-VDE passthrough path. NOTE: Full multi-shard federation routing requires a RoutingTable and VdeFederationRouter setup — if the constructor demands all these, create minimal instances. If too complex, simplify to proving that 2 independent VDEs can be created, populated, and operated without interference.

5. **TestPolicyCascadeThroughHierarchy**: Create a PolicyResolutionEngine (or the simplest available policy infrastructure). Set a policy at "global" scope (e.g., compression=enabled). Resolve the policy at a child path — verify it inherits the global setting. Set an override at the child path (e.g., compression=disabled). Resolve again — verify override takes precedence. This tests cascade semantics. If PolicyResolutionEngine requires complex setup (IPolicyStore, etc.), use CascadeOverrideStore with in-memory backing. If even that is too heavyweight, create a minimal test that verifies CascadeStrategies.ResolveWithOverrides works correctly with mock data (create the simplest possible path to prove cascade behavior).

6. **TestFullStackTeardownAndRebootstrap**: Create VDE on temp file. Store 5 objects. Dispose VDE (simulates shutdown). Create a NEW VirtualDiskEngine instance with same ContainerPath. Call InitializeAsync (re-reads superblock). Retrieve all 5 objects — verify data matches original. This proves persistence across VDE lifecycle (the container file survives teardown and re-bootstrap).

7. **TestDevicePoolMetadataConsistency**: Create a DevicePoolDescriptor with 4 members across 2 tiers (2 Hot, 2 Warm). Verify PoolId is non-empty GUID. Verify each member has unique DeviceId. Verify total capacity = sum of member capacities minus reserved. Create a CompoundBlockDevice from the "Hot" tier devices only (filter by MediaType or index). Write pattern, read back, verify. Then create a second CompoundBlockDevice from "Warm" tier. Write different pattern, read back, verify. This proves tiered pool organization works correctly.
