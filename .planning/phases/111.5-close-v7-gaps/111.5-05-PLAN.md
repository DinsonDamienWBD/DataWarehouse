Create 7 fuzz harness classes. Each class follows this pattern:

```csharp
using SharpFuzz;
using Xunit;

namespace DataWarehouse.Fuzz.Tests;

[Trait("Category", "Fuzz")]
public class Vde[Subsystem]FuzzTests
{
    /// <summary>
    /// Fuzz test that feeds corrupted bytes to [target method].
    /// Asserts no unhandled exceptions escape -- all corruption must be caught.
    /// </summary>
    [Fact]
    public void Fuzz_[TargetMethod]_HandlesCorruptInput()
    {
        // Generate a corpus of valid + corrupted inputs
        var validData = CreateValidInput();
        var corruptInputs = GenerateCorruptVariants(validData);

        foreach (var input in corruptInputs)
        {
            // Must not throw unhandled exceptions
            try
            {
                [TargetMethod](input);
            }
            catch (Exception ex) when (
                ex is VdeCorruptionException ||
                ex is InvalidDataException ||
                ex is FormatException ||
                ex is ArgumentException ||
                ex is ArgumentOutOfRangeException ||
                ex is OverflowException ||
                ex is IndexOutOfRangeException)
            {
                // Expected: corruption detected and reported
            }
            // Any other exception type = test failure (unhandled corruption)
        }
    }
}
```

**IMPORTANT**: Before writing each harness, READ the actual VDE source files to understand:
- The exact method signatures for Deserialize/Parse methods
- What exception types they throw on invalid input
- What parameters they accept (byte[], Span<byte>, Stream, etc.)
- Constructor requirements

For each harness, generate corrupted input variants:
- All zeros
- All 0xFF
- Valid header + truncated body
- Valid structure with flipped bits at random positions
- Maximum-value integers in size fields
- Negative values in signed size fields
- Zero-length data
- Extremely large claimed sizes (decompression bomb detection)
- Overlapping ranges / circular references where applicable

**7 harness files:**

1. **VdeBlockAllocationFuzzTests.cs** — Target: ExtentTree.LoadAsync (corrupt XxHash64 checksums, invalid extent counts), BitmapAllocator (corrupt bitmap bytes), Metaslab.EnsureBitmapLoadedAsync (corrupt bitmap deserialization). Use Span<byte> or byte[] inputs simulating corrupt block data.

2. **VdeAllocationGroupFuzzTests.cs** — Target: AllocationGroupDescriptorTable (corrupt AG descriptors with invalid offsets/counts), SubBlockPacker/SubBlockBitmap (corrupt sub-block metadata), ExtentTreeNode (corrupt child pointers).

3. **VdeCompressionFuzzTests.cs** — Target: PerExtentCompressor.DecompressExtentAsync (corrupt Brotli payloads, mismatched OriginalSize/CompressedSize), DataTransformerStage (corrupt pipeline context data). Include decompression bomb test: CompressedExtent claiming OriginalSize >> actual.

4. **VdeFormatParsingFuzzTests.cs** — Target: Superblock.Deserialize (corrupt magic, invalid blockSize, negative TotalBlocks, mismatched XxHash64), SuperblockV2.Deserialize (corrupt ModuleManifest, invalid InodeSize), MagicSignature.Deserialize, CompactInode64/InodeV2/Inode.Deserialize (corrupt type enums, negative sizes, invalid block pointers), VdeDetector.Detect (adversarial byte sequences), PreambleHeader.Deserialize. This is the largest harness -- 10+ test methods.

5. **VdeIntegrityFuzzTests.cs** — Target: HierarchicalChecksumTree.VerifyBlockChecksum (corrupt trailers), CrossExtentIntegrityChain (corrupt per-block records), BlockChecksummer/ChecksumTable (corrupt entries), UniversalBlockTrailer.Verify (corrupt payloads).

6. **VdeJournalFuzzTests.cs** — Target: JournalEntry deserialization (corrupt SequenceNumber, TransactionId, BeforeImage/AfterImage), WriteAheadLog/ShardedWriteAheadLog replay (corrupt log blocks), CheckpointManager recovery (corrupt checkpoint data), WalTransaction boundaries with interleaved corrupt entries.

7. **VdeCowDedupFuzzTests.cs** — Target: ExtentAwareCowManager.DeserializeRefCounts (corrupt ref count table -- verify existing guard holds), ContentAddressableDedup (corrupt content-address hash entries), SnapshotManager (corrupt inode references).

Each harness class should have 5-8 test methods, total 35-56 fuzz tests.

CRITICAL ASSERTIONS for every fuzz test:
- No unhandled exceptions escape (only expected exception types in catch)
- No infinite loops (use CancellationToken with 10-second timeout per input)
- Memory stays bounded: check GC.GetTotalMemory before/after, assert < 500MB growth
- Use try/finally to ensure any ArrayPool rentals are returned

If certain VDE classes don't have Deserialize methods or accept different input formats than expected, adapt the harness to match the actual API surface. Read the source files first.
