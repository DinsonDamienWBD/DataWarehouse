  <name>Task 1: Coyote 10K concurrency testing + Stryker all projects</name>
  <files>Metadata/final-hardening-report.md, Metadata/stryker-final-report/surviving-mutants.md</files>
  <action>
Execute concurrency and mutation testing:

1. **Coyote 10,000-iteration testing**:
   - Identify all concurrent code paths: classes using `lock`, `SemaphoreSlim`, `ConcurrentDictionary`, `Interlocked`, `async`/`await` with shared state, `Channel<T>`, `TaskCompletionSource`.
   - Run Coyote with `--iterations 10000` on each concurrent scenario:
     - VDE write/read under concurrent access
     - Message bus concurrent publish/subscribe
     - Strategy resolution under concurrent requests
     - Cache eviction under concurrent access (BoundedCache)
     - WAL group commit under concurrent writers
     - Federation router under concurrent requests
   - If Coyote finds a deadlock or race: fix it, re-run to confirm fix, document the bug and fix.

2. **Stryker mutation testing on all projects**:
   - Run `dotnet stryker` on each project in the solution (SDK, Kernel, key plugins).
   - For each surviving mutant:
     - Analyze if it is an equivalent mutant (mutation produces identical behavior).
     - If equivalent: document in `Metadata/stryker-final-report/surviving-mutants.md` with explanation.
     - If killable: write a test that kills it, add to appropriate test project.
   - Target: >=90% mutation score across solution, >=85% per project.

3. **Document results** in `Metadata/final-hardening-report.md` Section 1 "Coyote Concurrency Results" and Section 2 "Stryker Mutation Results".

Note: Stryker and Coyote runs are time-intensive. Prioritize SDK and Kernel projects first, then high-risk plugins (Security, VDE, Federation). If time-constrained, document which projects were tested.
  </action>
