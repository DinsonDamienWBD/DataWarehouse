<objective>
Create SpdkCompoundDeviceAdapter to bridge SpdkBlockDevice to IPhysicalBlockDevice, enabling SPDK as a backing device for CompoundBlockDevice RAID configurations while maintaining DMA alignment through parity calculations.

Purpose: CompoundBlockDevice implements RAID 0/1/5/6/10 over IPhysicalBlockDevice instances. SpdkBlockDevice implements IDirectBlockDevice, not IPhysicalBlockDevice. An adapter bridges this gap so SPDK-backed RAID arrays deliver the 10 GB/s sequential, 2M+ IOPS random 4K targets.

Output: SpdkCompoundDeviceAdapter + validation tests for SPDK-backed RAID configurations.
</objective>

First read `DataWarehouse.SDK/VirtualDiskEngine/PhysicalDevice/IPhysicalBlockDevice.cs` to understand the full interface contract.

Create `SpdkCompoundDeviceAdapter.cs`:
1. `public sealed class SpdkCompoundDeviceAdapter : IPhysicalBlockDevice`
2. Constructor takes `SpdkBlockDevice innerDevice, string deviceId`
3. Implement all `IPhysicalBlockDevice` properties:
   - Map `BlockSize`, `BlockCount` from inner device
   - `DeviceId` from constructor parameter (typically the PCI address)
   - `IsHealthy` — true while inner device is not disposed
   - `LatencyMicroseconds` — report SPDK nominal latency (0.5 us)
   - `ThroughputMBps` — report from device capability if probed, else estimate (3000 MB/s typical NVMe)
   - Other physical properties: derive sensible defaults for NVMe
4. Delegate `ReadBlockAsync`, `WriteBlockAsync`, `FlushAsync`, `DisposeAsync` to inner device
5. **DMA alignment preservation**: When CompoundBlockDevice does RAID parity calculation, it reads data into buffers, XORs them, and writes parity. The adapter ensures buffers passed to SpdkBlockDevice remain DMA-aligned by:
   - Providing `GetAlignedBuffer` that allocates from SPDK DMA pool
   - Documenting that CompoundBlockDevice callers SHOULD use this for parity buffers
6. Mark with `[SdkCompatibility("8.0.0", Notes = "Phase 112: SPDK to IPhysicalBlockDevice adapter for RAID")]`
