    - Test: SpdkDmaBufferPool tracks outstanding count correctly

  <action>
1. Modify `SpdkDmaAllocator.cs`:
   - Change class declaration to `internal sealed class SpdkDmaAllocator : IMemoryAllocator`
   - Add `public int Alignment => DefaultAlignment;` property
   - Add parameterless `IMemoryOwner<byte> Allocate(int byteCount)` that delegates to existing `Allocate(byteCount, DefaultAlignment)`
   - Rename existing 2-parameter `Allocate` to `AllocateAligned` to avoid ambiguity
   - Add `IDisposable` implementation — track allocated count via `Interlocked.Increment/Decrement`, log warning if count > 0 on dispose
   - Update `SpdkDmaMemoryOwner.Dispose` to decrement the parent allocator's outstanding count
   - Add `[SdkCompatibility("8.0.0", Notes = "Phase 112: IMemoryAllocator conformance")]` attribute

2. Create `SpdkDmaBufferPool.cs`:
   - `internal sealed class SpdkDmaBufferPool : IDisposable`
   - Constructor takes `SpdkDmaAllocator allocator, int bufferSize, int initialPoolSize = 32`
   - Pre-allocates `initialPoolSize` DMA buffers into a `ConcurrentStack<IMemoryOwner<byte>>`
   - `Rent()` method: tries pop from stack, falls back to `_allocator.Allocate(bufferSize)` on empty pool
   - `Return(IMemoryOwner<byte>)` method: pushes back to stack (zero-fills first for security)
   - Track outstanding via `_outstandingCount` (Interlocked)
   - `Dispose()`: drain pool and dispose all buffers, log warning if outstanding > 0 using `ILogger` if provided
   - Mark with `[SdkCompatibility("8.0.0", Notes = "Phase 112: DMA buffer pool for SPDK I/O path")]`

3. Update `SpdkBlockDevice.cs` to use `SpdkDmaBufferPool` instead of direct `SpdkDmaAllocator.Allocate` calls in hot paths (ReadBlockAsync, WriteBlockAsync, ReadBatchAsync, WriteBatchAsync). Initialize pool in constructor with `_queueDepth` buffers.
