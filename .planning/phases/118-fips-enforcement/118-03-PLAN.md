Create `docs/security/FIPS-140-Security-Policy.md` — a FIPS 140-2/3 Security Policy template following CMVP IG (Implementation Guidance) structure:

Sections:
1. **Module Overview** — DataWarehouse Cryptographic Module, boundary definition (software module), operational environment (.NET 8+), module type (software)
2. **Security Level** — Target: FIPS 140-3 Level 1 (software), justify why Level 1 is appropriate
3. **Cryptographic Module Ports and Interfaces** — Data input, data output, control input, status output (logical interfaces)
4. **Roles, Services, and Authentication** — Crypto Officer role (admin), User role, services per role (key generation, encryption, signing, etc.)
5. **Finite State Model** — States: Power-Off, Self-Test, Operational, Error. Transitions between states.
6. **Physical Security** — N/A for Level 1 software module, reference to deployment environment requirements
7. **Operational Environment** — .NET 8+ runtime, Windows/Linux, FIPS mode enforcement via `FipsModeEnforcer`
8. **Cryptographic Key Management** — Reference to Key Lifecycle doc, key zeroization requirements
9. **Self-Tests** — Power-up self-tests (algorithm KATs), conditional self-tests (pair-wise consistency)
10. **Design Assurance** — Configuration management, secure delivery, testing methodology
11. **Mitigation of Other Attacks** — Side-channel mitigations, constant-time operations

  <action>
Create `docs/security/FIPS-Key-Lifecycle.md` following NIST SP 800-57 Part 1 key management framework:

Sections:
1. **Key Types** — Table of all key types used by DataWarehouse: VDE master keys, data encryption keys (DEK), key encryption keys (KEK), HMAC keys, signing keys, transit encryption keys. For each: algorithm, key length, usage, max cryptoperiod.
2. **Key Generation** — FIPS-approved RNG (CTR-DRBG via .NET), seed source requirements, entropy requirements (SP 800-90B)
3. **Key Storage** — At rest: encrypted in key store (HSM-backed if available, software key wrapping otherwise). Key wrapping using AES-256-KW (SP 800-38F). ConfigurationItem<bool> HsmRequired.
4. **Key Distribution** — Key transport via RSA-OAEP or ECDH key agreement. TLS 1.2+ for in-transit key material.
5. **Key Usage** — Separation of key types (signing vs encryption), usage limits per SP 800-57, domain separation
6. **Key Rotation** — Rotation intervals per CipherPreset (7-365 days), automated rotation trigger, rotation without downtime
7. **Key Archival** — Archive requirements for audit/compliance, encrypted archive format
8. **Key Destruction** — Zeroization requirements (SP 800-88), memory scrubbing (`CryptographicOperations.ZeroMemory`), secure erase of key material
9. **Key Recovery** — Split-key reconstruction, M-of-N threshold schemes, recovery audit trail
10. **Compliance Mapping** — Table mapping FIPS 140-3 key management requirements to DataWarehouse implementation (requirement ID, DW feature, status)

Each section should reference the specific DataWarehouse SDK class/method that implements the requirement.
