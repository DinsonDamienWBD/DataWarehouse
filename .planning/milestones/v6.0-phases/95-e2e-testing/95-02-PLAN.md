**Required tests:**

1. **TestMultiVdeIsolationAsync** — Cross-VDE data isolation:
   - Create 2 separate device pools (4 devices each)
   - Build 2 CompoundBlockDevices (RAID 5 each)
   - Create 2 VDE instances (vdeA, vdeB) on separate compound devices (separate temp files)
   - Initialize both
   - Store "shared-key" with data-A in vdeA
   - Store "shared-key" with data-B in vdeB (same key name, different data)
   - Retrieve "shared-key" from vdeA -> verify data-A
   - Retrieve "shared-key" from vdeB -> verify data-B (isolation confirmed)
   - List vdeA: verify only vdeA's objects visible
   - List vdeB: verify only vdeB's objects visible
   - Dispose both VDEs

2. **TestDeviceLevelRaid6AcrossPoolAsync** — Device-level RAID 6 with dual failure:
   - Create 8 InMemoryPhysicalBlockDevice (4096 block size, 16384 blocks each)
   - Build CompoundBlockDevice with RAID 6 (6 data + 2 parity)
   - Create VDE, initialize
   - Store 10 objects (each 8KB)
   - Take device[0] offline, verify all 10 objects retrievable
   - Take device[1] offline (2 failures), verify all 10 objects still retrievable
   - Take device[2] offline (3 failures - beyond RAID 6 tolerance)
   - Attempting to retrieve should fail or throw (RAID 6 cannot survive 3 failures)
   - Wrap the 3-failure test in try/catch, assert it throws IOException or InvalidOperationException
   - Restore all devices, verify data accessible again

3. **TestDualRaidSimultaneousFailureAsync** — Device RAID 6 + data erasure coding:
   - Create 8 devices, build CompoundBlockDevice with RAID 6
   - Create VDE on top
   - Store objects that exercise both RAID levels
   - Fail 1 device at device level (RAID 6 absorbs it)
   - Verify all data accessible
   - The data-level RAID (if VDE uses it internally for data blocks) provides a second domain
   - Assert: device-level and data-level RAID are independent — a device failure does not cascade into data-level RAID failure
   - Assert CompoundBlockDevice health shows Degraded state
   - Store additional data in degraded mode — succeeds
   - Retrieve additional data — succeeds

4. **TestCrossVdeOperationsAsync** — Operations spanning multiple VDEs:
   - Create 3 VDEs on separate compound devices (3 devices each, RAID 5)
   - Store different data sets in each (5 objects per VDE, total 15)
   - Verify each VDE's list returns only its own 5 objects
   - Delete 2 objects from VDE-2
   - Verify VDE-1 and VDE-3 unaffected (still 5 objects each)
   - Verify VDE-2 has 3 objects

5. **TestTBScaleSimulationAsync** — TB-scale multi-VDE:
   - Create 4 VDEs, each backed by 6-device RAID 6 (each device 32768 blocks = 128MB)
   - Total addressable space: 4 * 4 * 128MB = 2GB (representing TB-scale architecture)
   - Store 100 objects across all 4 VDEs (25 per VDE), varying sizes 1KB-100KB
   - List all objects per VDE, verify counts
   - Retrieve random sample (5 from each VDE) and verify data integrity
   - Delete 10 objects from VDE-1, verify count goes to 15

6. **TestHotSpareRebuildUnderLoadAsync** — Hot spare rebuild during active I/O:
   - Create 6 data devices + 1 spare (7 total), RAID 6 compound device
   - Create HotSpareManager with policy (1 spare, auto-rebuild=true)
   - Create VDE, store 10 objects
   - Fail device[0] via SetOnline(false)
   - While rebuild would occur (conceptually), continue storing 5 more objects
   - Verify all 15 objects retrievable
   - Assert HotSpareManager state reflects the spare was activated or rebuild was initiated

Each test uses try/finally for cleanup (disposing VDEs, deleting temp files).
