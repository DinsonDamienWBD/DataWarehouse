Create `DataWarehouse.SDK.VirtualDiskEngine.E2E.E2ETestInfrastructure` as a static class providing shared helpers for all E2E tests.

Use `[SdkCompatibility("6.0.0")]` on the class.

**Required helpers:**

1. `CreateInMemoryDevices(int count, int blockSize = 4096, long blockCount = 65536)` — Creates an array of `InMemoryPhysicalBlockDevice` instances. Import from `DataWarehouse.Plugins.UltimateRAID.Strategies.DeviceLevel`. NOTE: Since SDK cannot reference plugins, create a LOCAL `InMemoryPhysicalBlockDevice` in the E2E namespace that implements `IPhysicalBlockDevice` (follow the pattern from `Federation/Tests/FederationIntegrationTests.cs` which has a local `InMemoryBlockDevice`). This local version MUST implement full `IPhysicalBlockDevice` including TRIM, scatter-gather, `GetHealthAsync`, and `SetOnline(bool)` for failure injection. Use ConcurrentDictionary sparse storage like the plugin version.

2. `CreateCompoundDevice(IPhysicalBlockDevice[] devices, DeviceLayoutType layout, int stripeSizeBlocks = 256)` — Creates a `CompoundBlockDevice` via `CompoundDeviceConfiguration` and the `CompoundBlockDevice` constructor. This exercises the device pool -> compound device path.

3. `CreateVdeOnDevice(IBlockDevice device, string testName)` — Creates a VDE backed by the given block device. Since `VirtualDiskEngine` takes `VdeOptions` with a `ContainerPath`, and the E2E test needs to use an in-memory block device instead of a file, create a `CreateVdeWithTempFile(string testName, int blockSize = 4096, long totalBlocks = 65536)` that creates a temp file path and returns `(VirtualDiskEngine vde, string tempPath)`. The VDE uses the real file-backed container. Alternatively, if VirtualDiskEngine supports an IBlockDevice constructor overload, use that. Check the VirtualDiskEngine constructor — if it only supports ContainerPath, use temp files with cleanup.

4. `Assert(bool condition, string message)` — Standard assertion helper (throw InvalidOperationException on failure).

5. `AssertDeviceHealth(IPhysicalBlockDevice device)` — Calls `GetHealthAsync`, asserts temperature >= 0, wearLevel in [0,100], errorsUncorrectable >= 0.

6. `GenerateTestData(int sizeBytes)` — Returns a byte[] filled with a deterministic pattern (e.g., block index XOR'd with seed) for round-trip verification.

7. `VerifyDataRoundTrip(byte[] original, Stream retrieved)` — Reads stream to byte[], compares with original, asserts equality.

The namespace is `DataWarehouse.SDK.VirtualDiskEngine.E2E`.

<task type="auto">
  <name>Task 2: Single-VDE E2E Tests</name>
  <files>DataWarehouse.SDK/VirtualDiskEngine/E2E/SingleVdeE2ETests.cs</files>
  <action>
Create `DataWarehouse.SDK.VirtualDiskEngine.E2E.SingleVdeE2ETests` as a static class with `RunAllAsync(CancellationToken)` returning `(int passed, int failed, IReadOnlyList<string> failures)` — same pattern as `FederationIntegrationTests`.

Use `[SdkCompatibility("6.0.0", Notes = "Phase 95: Single-VDE E2E Tests")]`.

**Required tests (each is an isolated async method):**

1. **TestBareMetalBootstrapAsync** — The core bare-metal-to-user flow:
   - Create 4 InMemoryPhysicalBlockDevice (4096 block size, 16384 blocks each = 64MB each)
   - Build CompoundBlockDevice with RAID 5 layout (3 data + 1 parity)
   - Assert CompoundBlockDevice.BlockSize == 4096 and BlockCount is correct for RAID 5 (3 * 16384)
   - Create VDE on top (use temp file container with matching block size / total blocks from compound)
   - Initialize VDE
   - Store 3 objects with different sizes (1KB, 64KB, 1MB) using `StoreAsync`
   - Retrieve each and verify data matches using `VerifyDataRoundTrip`
   - List all objects and verify count == 3
   - Delete one object, list again and verify count == 2
   - Dispose VDE and compound device

2. **TestDeviceFailureInjectionAsync** — Device failure during RAID operation:
   - Create 6 InMemoryPhysicalBlockDevice, build CompoundBlockDevice with RAID 6 (double parity)
   - Create VDE, initialize, store 5 objects
   - Take 1 device offline via `SetOnline(false)` on the local InMemoryPhysicalBlockDevice
   - Verify all 5 objects still retrievable (RAID 6 tolerates 1 failure)
   - Store 1 more object (write succeeds in degraded mode)
   - Verify new object retrievable
   - Take a 2nd device offline
   - Verify all 6 objects still retrievable (RAID 6 tolerates 2 failures)
   - Restore both devices via `SetOnline(true)`
   - Verify all 6 objects still retrievable

3. **TestSmartHealthVerificationAsync** — SMART health monitoring:
   - Create 4 InMemoryPhysicalBlockDevice
   - For each device, call `GetHealthAsync`
   - Assert each returns valid PhysicalDeviceHealth (temperature >= 0, no uncorrectable errors initially)
   - Write data to force some I/O (via CompoundBlockDevice write)
   - Re-check health: total bytes written should increase
   - Assert `DeviceInfo.SerialNumber` is not null/empty for each device

4. **TestCrudOperationsFullCycleAsync** — Comprehensive CRUD:
   - Create 3 devices, RAID 5 CompoundBlockDevice, VDE
   - Store 10 objects with keys "obj-000" through "obj-009", each 4KB
   - List with prefix "obj-00" and verify 10 results
   - Retrieve each by key and verify data integrity
   - Delete "obj-003", "obj-007"
   - List again and verify 8 results, verify deleted keys absent
   - Store "obj-003" again with different data
   - Retrieve "obj-003" and verify it returns the NEW data, not the old
   - List and verify count == 9

5. **TestSingleVdeZeroFederationOverheadAsync** — Prove no federation:
   - Create a single CompoundBlockDevice + VDE
   - Store and retrieve 1 object
   - Verify: no RoutingTable, no ShardBloomFilterIndex, no CrossShardOperationCoordinator involved
   - This is a structural assertion: the VDE is directly on top of CompoundBlockDevice with no federation wrapping
   - Assert CompoundBlockDevice.Devices.Count matches expected device count
   - Assert VDE operations complete without any shard-related exceptions

6. **TestLargeObjectStorageAsync** — GB-scale simulation:
   - Create 8 devices (4096 blocks, 32768 blocks each = 128MB each), RAID 6
   - Create VDE
   - Store 1 object of 10MB (2560 blocks)
   - Retrieve and verify first 4096 bytes and last 4096 bytes match
   - Store 50 small objects (1KB each)
   - List all 51 objects
   - Delete the 10MB object
   - List and verify 50 objects remain

Each test follows the pattern: setup -> exercise -> verify -> teardown (dispose). Use try/finally for disposal.
