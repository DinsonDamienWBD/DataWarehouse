1. **Test100PlusShardFederationAsync** — PB-scale shard count:

   - Fan-out Search for "obj-5": verify returns 128 results (one per shard)

   - Assert bloom filter integration: create ShardBloomFilterIndex, add all keys, verify FilterCandidateShards returns correct narrow set for a specific key

2. **TestHierarchicalCatalogThreeLevelAsync** — SuperFederation with 3 levels:
   - Level 1 (top): SuperFederationRouter with 4 region prefixes ("us-east/", "us-west/", "eu-west/", "ap-south/")
   - Level 2: 4 FederationNodes, each containing its own router with 8 shards (32 shards total)
   - Level 3: 32 LeafVdeNodes
   - Resolve paths through all 3 levels:
     - "us-east/users/alice" -> must resolve to a us-east leaf shard
     - "eu-west/orders/123" -> must resolve to a eu-west leaf shard
   - Build FederationTopology and verify: 1 super-federation, 4 federations, 32 leafs, 37 total nodes
   - Verify deterministic routing: same key always resolves to same leaf

3. **TestFanOutListSearchMergeSortedAsync** — Merge-sorted fan-out:
   - Create 5 shards with interleaved keys:
     - Shard 0: "alpha", "echo", "india"
     - Shard 1: "bravo", "foxtrot", "juliet"
     - Shard 2: "charlie", "golf", "kilo"
     - Shard 3: "delta", "hotel", "lima"
     - Shard 4: "mike", "november", "oscar"
   - Fan-out List: verify all 15 results in alphabetical order
   - Fan-out Search "o": verify results matching "o" substring in sorted order (bravo, foxtrot, golf, hotel, november, oscar)
   - Fan-out List with limit=5: verify returns first 5 alphabetically
   - Delete "charlie" from shard 2
   - Fan-out List: verify 14 results, "charlie" absent

4. **TestShardSplitMergeUnderLoadAsync** — Split/merge during active reads:
   - Create 4 shards (64 slots, 16 per shard), populate 20 objects per shard (80 total)
   - Create ShardMigrationEngine with routing table, redirect table, lock service
   - Start a "background load" concept: store the initial state
   - Migrate shard-0 slots (0..16) to a new shard-4 (simulates split)
   - After migration completes, verify:
     - Routing table slots 0..15 now point to shard-4
     - Routing table slots 16..63 unchanged
     - Fan-out List still returns all 80 objects (data migrated to shard-4)
   - Verify migration state reached Completed
   - Verify MigrationRedirectTable has no active migrations after completion

5. **TestNamespaceTransparencyAsync** — Identical API at all scales:
   - Phase A (single-shard): Create 1 shard, store "doc/readme", retrieve, list, delete
   - Phase B (multi-shard): Create 10 shards, same operations via CrossShardOperationCoordinator
   - Phase C (federated): Create SuperFederationRouter wrapping 2 federations of 5 shards each, same operations
   - Verify: the API signatures are identical across all three scales (FanOutListAsync, FanOutSearchAsync, FanOutDeleteAsync)
   - Verify: results are semantically identical (same keys, same counts)
   - This proves transparent namespace: callers don't know whether they're hitting 1 shard or 100

6. **TestPolicyCascadeThroughFederationAsync** — Policy propagation:
   - Create a 3-level federation (super -> 2 feds -> 8 shards total)
   - Create FederationOptions with ShardOperationTimeout=30s, MaxConcurrentShardOperations=4
   - Create CrossShardOperationCoordinator with these options
   - Verify operations respect the timeout (use a very short timeout to trigger timeout behavior)
   - Create ShardTransactionCoordinator with the same options
   - Begin a transaction across 3 shards with 1ms timeout
   - Wait for timeout, verify transaction is cleaned up
   - This proves policy (timeouts, concurrency limits) cascades through the federation

7. **TestBareMetalToServingFullStackAsync** — Complete bare-metal bootstrap:
   - Step 1: Create 16 InMemoryPhysicalBlockDevice (use E2ETestInfrastructure helpers or local helpers)
   - Step 2: Group into 2 pools of 8 devices
   - Step 3: Build 2 CompoundBlockDevices with RAID 6 (6 data + 2 parity each)
   - Step 4: Create 2 VDE instances on the compound devices
   - Step 5: Create InMemoryShardVdeAccessor wrapping both VDEs (simulated — add objects directly)
   - Step 6: Create VdeFederationRouter with routing table (128 slots, 64 per shard)
   - Step 7: Create CrossShardOperationCoordinator
   - Step 8: Store objects through the coordinator
   - Step 9: Fan-out List to verify all objects visible
   - Step 10: Fail 1 device in pool-1, verify data still accessible through federation
   - Assert: raw devices -> pools -> compound devices -> VDEs -> federation -> user operations all work end-to-end
