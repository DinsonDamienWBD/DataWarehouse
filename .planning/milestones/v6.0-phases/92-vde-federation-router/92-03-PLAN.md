- `public sealed class ShardBloomFilter`
- Constructor: `ShardBloomFilter(int filterSizeBytes = 8192, int hashFunctionCount = 10)` -- 8KB filter with 10 hash functions gives ~0.1% FPR at ~5000 keys (well under 1% target for 99%+ rejection)
- Follow the same XxHash64 pattern as `TagBloomFilter`: hash key bytes with K different seeds (seed = hashIndex * 0x9E3779B9), set/test bits

- `public bool MayContain(ReadOnlySpan<byte> keyUtf8)` -- returns true if possibly present, false if definitely absent
- `public bool MayContain(string key)` -- convenience overload
- `public int FilterSizeBytes { get; }`
- `public int HashFunctionCount { get; }`
- `public long EstimatedElementCount { get; private set; }` -- incremented on Add
- Binary serialization: `WriteTo(Span<byte> dest)` writes filterSizeBytes(4) + hashFunctionCount(4) + estimatedCount(8) + bits. `static ReadFrom(ReadOnlySpan<byte>)` reads back. `public int SerializedSize => 16 + FilterSizeBytes;`
- `public long ComputeDigest()` -- compute 8-byte XxHash64 digest of the entire filter bits array for embedding in CatalogEntry.BloomFilterDigest (quick check: if digest unchanged, no need to reload full filter)

**ShardCatalogResolver.cs** -- implements `IShardCatalogResolver` (from 92-01 VdeFederationRouter.cs):
- `[SdkCompatibility("6.0.0", Notes = "Phase 92: VDE 2.0B Catalog Resolver (VFED-13)")]`
- `public sealed class ShardCatalogResolver : IShardCatalogResolver`
- Constructor: `ShardCatalogResolver(RootCatalog rootCatalog, DomainCatalogCache domainCatalogCache, Func<Guid, CancellationToken, ValueTask<DomainCatalog?>> domainLoader, Func<Guid, CancellationToken, ValueTask<IndexShardCatalog?>> indexShardLoader)`
  - `domainLoader`: async factory that loads a DomainCatalog from its VDE (for cache misses)
  - `indexShardLoader`: async factory that loads an IndexShardCatalog from its VDE (index shards are not cached -- they are lightweight and accessed via bloom filter fast rejection)
- Private fields: `_rootCatalog`, `_domainCache`, `_domainLoader`, `_indexShardLoader`

- **Core method**: `public async ValueTask<ShardAddress> ResolveAsync(string key, ReadOnlyMemory<int> pathSlots, CancellationToken ct = default)`
  Implementation (at most FederationConstants.MaxColdPathHops iterations):
  ```
  Hop 1: domainSlot = pathSlots.Span[0]
         rootEntry = _rootCatalog.LookupDomain(domainSlot)
         if null -> return ShardAddress.None

  Hop 2: domainCatalog = _domainCache.Get(rootEntry.ShardVdeId)
         if null -> domainCatalog = await _domainLoader(rootEntry.ShardVdeId, ct)
                    if null -> return ShardAddress.None
                    _domainCache.Put(rootEntry.ShardVdeId, domainCatalog)
         indexSlot = pathSlots.Length > 1 ? pathSlots.Span[1] : pathSlots.Span[0]
         indexEntry = domainCatalog.LookupIndexShard(indexSlot)
         if null -> return ShardAddress.None

  Hop 3: indexShard = await _indexShardLoader(indexEntry.ShardVdeId, ct)
         if null -> return ShardAddress.None
         dataShard = indexShard.LookupDataShard(key)
         if null -> return ShardAddress.None (bloom filter rejected)

  Return: new ShardAddress(dataShard.DataVdeId, 0, ShardLevel.Data)
  ```
  - The InodeNumber in ShardAddress is 0 at this stage -- the actual inode is resolved by the data VDE itself when the operation reaches it. The federation router just needs to know WHICH VDE to send the operation to.
  - Track hop count with a counter, throw `InvalidOperationException($"Federation resolution exceeded {FederationConstants.MaxColdPathHops} hops")` if exceeded.

- `public CatalogResolutionStats GetStats()` -- track per-level hit/miss counts
- `public readonly record struct CatalogResolutionStats(long RootHits, long RootMisses, long DomainCacheHits, long DomainCacheMisses, long IndexShardHits, long IndexBloomRejections);`
- Use `Interlocked.Increment` for all stat counters (lock-free hot path)
