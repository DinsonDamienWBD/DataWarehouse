<objective>
Integrate bloom filters at the index shard level for negative lookup rejection. Each index shard maintains a bloom filter of its keys; the federation router queries these filters to skip shards that definitely do not contain a requested key, avoiding unnecessary I/O.

Purpose: Without bloom filters, every Retrieve/Exists operation in a federated deployment must check all index shards. Bloom filters eliminate 99%+ of negative lookups, keeping federated read latency close to single-VDE latency.

Output: ShardBloomFilterIndex.cs, BloomFilterPersistence.cs
</objective>

This class manages a collection of bloom filters, one per index shard, for fast negative lookups in the federation layer.

Constructor: `ShardBloomFilterIndex(ShardBloomFilterConfig config)`

Internal state:
- `private readonly ConcurrentDictionary<Guid, BloomFilter<string>> _shardFilters` — maps ShardVdeId to its bloom filter. Use the SDK `BloomFilter<string>` from `DataWarehouse.SDK.Primitives.Probabilistic` (NOT a custom implementation). Pass serializer `key => System.Text.Encoding.UTF8.GetBytes(key)`.
- `private readonly ShardBloomFilterConfig _config`

Methods:

1. `bool MayContain(Guid shardVdeId, string key)`:
   - If no filter exists for shardVdeId, return true (conservative — might contain).
   - Call `_shardFilters[shardVdeId].MayContain(key)`.
   - Return result. True = "might be here" (check data shard). False = "definitely not here" (skip shard).

2. `IReadOnlyList<Guid> FilterCandidateShards(IReadOnlyList<Guid> allShards, string key)`:
   - For each shard, call MayContain. Return only shards that may contain the key.
   - If result is empty (all filters say no), return all shards as fallback (defensive against total bloom filter corruption).
   - This is the method VdeFederationRouter calls before routing a lookup.

3. `void Add(Guid shardVdeId, string key)`:
   - Get or create filter for shard (lazy initialization with config.ExpectedKeysPerShard and config.TargetFalsePositiveRate).
   - Call `filter.Add(key)`.
   - If EnableAutoRebuild and filter.CurrentFalsePositiveRate > config.TargetFalsePositiveRate * 2, mark shard for rebuild (set flag, don't rebuild inline).

4. `void Remove(Guid shardVdeId, string key)`:
   - Bloom filters cannot remove items. Mark the shard's filter as "dirty" (needs rebuild).
   - Track delete count per shard. When delete count exceeds 10% of expected keys, trigger rebuild flag.

5. `Task RebuildFilterAsync(Guid shardVdeId, IAsyncEnumerable<string> allKeys, CancellationToken ct = default)`:
   - Create fresh BloomFilter<string> with config parameters.
   - Iterate allKeys, adding each to the new filter.
   - Atomically swap: `_shardFilters[shardVdeId] = newFilter`.
   - Reset delete counter for shard.

6. `void RemoveShard(Guid shardVdeId)`:
   - Remove filter from dictionary. Called when shard is decommissioned.

7. `BloomFilterSnapshot GetSnapshot(Guid shardVdeId)`:
   - Returns current filter state for persistence. Define:
   ```csharp
   public readonly record struct BloomFilterSnapshot(Guid ShardVdeId, byte[] FilterData, long ItemCount, double FalsePositiveRate);
   ```
   - FilterData is obtained by accessing the BloomFilter's bit array. Since BloomFilter<T> from SDK Primitives may not expose raw bits directly, use reflection or add a `GetBits()` method. If BloomFilter<T> has Serialize/Deserialize pattern, use that. Otherwise, store the entire filter via MemoryMarshal or a serialization helper.
   - IMPORTANT: Check the actual BloomFilter<T> API in the SDK. If it already has Serialize/ToBytes methods, use them. If not, create a minimal helper that accesses the internal BitArray.

8. `void LoadSnapshot(BloomFilterSnapshot snapshot)`:
   - Reconstruct filter from snapshot data and insert into _shardFilters.

Property: `IReadOnlyCollection<Guid> TrackedShards => _shardFilters.Keys`
Property: `int ShardCount => _shardFilters.Count`

DisposeAsync: Clear all filters, release memory.

Key design decisions:
- Use existing SDK BloomFilter<string> (DO NOT reimplement bloom filter logic).
- Lazy initialization means single-VDE deployments never allocate filters.
- Conservative fallback: if all filters reject, return all shards (prevents data loss from bloom filter bugs).
- Thread-safe via ConcurrentDictionary for filters, but individual BloomFilter<T> is NOT thread-safe per SDK docs — document that Add must be externally synchronized per shard (federation router serializes writes per shard via WAL transactions).

```csharp
[SdkCompatibility("6.0.0")]
public sealed class BloomFilterPersistence
```

Persists bloom filter snapshots to shard metadata blocks so filters survive VDE restart without full rebuild.

Constructor: `BloomFilterPersistence(int blockSize = 4096)`

On-disk format (header at start of first block):
- Bytes 0-3: Magic `0x424C4F4D` ("BLOM") as uint32 LE via BinaryPrimitives
- Bytes 4-7: Version uint32 LE (currently 1)
- Bytes 8-15: ShardVdeId first 8 bytes of Guid
- Bytes 16-23: ShardVdeId last 8 bytes of Guid
- Bytes 24-31: ItemCount int64 LE
- Bytes 32-39: FalsePositiveRate as double (BitConverter.Int64BitsToDouble pattern via BinaryPrimitives)
- Bytes 40-43: FilterDataLength int32 LE
- Bytes 44-47: Checksum uint32 LE (XxHash32 of FilterData, or simple CRC32 if XxHash not available)
- Bytes 48+: FilterData bytes (may span multiple blocks)

Methods:

1. `int CalculateBlocksNeeded(int filterDataLength)`:
   - Header is 48 bytes. First block holds (blockSize - 48) bytes of filter data.
   - Remaining data: `Math.Max(0, filterDataLength - (blockSize - 48))` bytes, rounded up to block boundaries.
   - Return total blocks: `1 + (int)Math.Ceiling((double)Math.Max(0, filterDataLength - (blockSize - 48)) / blockSize)`

2. `async Task WriteAsync(IBlockDevice device, long startBlock, BloomFilterSnapshot snapshot, CancellationToken ct = default)`:
   - Allocate byte array for header block (blockSize).
   - Write magic, version, Guid bytes (use `snapshot.ShardVdeId.ToByteArray()` and copy 16 bytes), item count, false positive rate, filter data length, checksum — all via BinaryPrimitives.WriteXXXLittleEndian.
   - Copy as much FilterData as fits into remainder of first block.
   - Write first block via `device.WriteBlockAsync(startBlock, ...)`.
   - For overflow data, write subsequent blocks (startBlock + 1, startBlock + 2, ...).
   - Pad final block to blockSize with zeros.

3. `async Task<BloomFilterSnapshot?> ReadAsync(IBlockDevice device, long startBlock, CancellationToken ct = default)`:
   - Read first block.
   - Verify magic bytes. If mismatch, return null (no filter persisted here).
   - Parse header fields via BinaryPrimitives.ReadXXXLittleEndian.
   - Reconstruct Guid from 16 bytes.
   - Read FilterData from remainder of first block + subsequent blocks as needed.
   - Verify checksum. If mismatch, return null (corrupted filter, will be rebuilt).
   - Return BloomFilterSnapshot.

4. `async Task EraseAsync(IBlockDevice device, long startBlock, int blockCount, CancellationToken ct = default)`:
   - Zero out blocks (write zero-filled blocks). Used when decommissioning a shard.

Use only BinaryPrimitives for all integer/long serialization. Use ArrayPool<byte>.Shared for temporary buffers. No BinaryReader/BinaryWriter. Validate all reads (bounds checking on FilterDataLength — must be non-negative and <= reasonable max like 64MB).

Checksum: Use `System.IO.Hashing.XxHash32` if available in the project's target framework. If not, use a simple additive checksum: sum all FilterData bytes as uint32 with unchecked overflow. Document which approach is used.
