<objective>
Implement recursive VDE 2.0B+ composition: a super-federation that treats VDE 2.0B instances as routing targets, enabling yottabyte-to-brontobyte scale through hierarchical nesting. Provide admin-visible topology inspection.

Purpose: A single VDE 2.0B federation has a practical shard limit. To scale beyond that, VDE 2.0B+ nests federations recursively — each node in the super-federation's routing table can be either a leaf VDE 2.0A instance or another VDE 2.0B federation. This enables unbounded horizontal scaling.

Output: SuperFederationRouter.cs, FederationTopology.cs
</objective>

```csharp
[SdkCompatibility("6.0.0")]
public sealed class SuperFederationRouter : IFederationNode
```
- The top-level recursive router. Its routing table maps namespace prefixes to IFederationNode children (which can be FederationNode or another SuperFederationRouter).
- Constructor: `SuperFederationRouter(Guid superFederationId)`

Internal state:
- `private readonly SortedList<string, IFederationNode> _routingTable` — sorted by namespace prefix for binary search (longest-prefix match).
- `private readonly Dictionary<Guid, IFederationNode> _nodesByIdCache` — quick lookup by node ID.
- `private int _depth = 0`

Properties:
- `Guid NodeId { get; }` — the super-federation's identity.
- `FederationNodeType NodeType => FederationNodeType.SuperFederation`
- `int Depth => _depth`

Methods:

1. `void AddNode(string namespacePrefix, IFederationNode node)`:
   - Adds a child node (federation or leaf) to the routing table under the given namespace prefix.
   - Validates prefix is non-empty and not duplicate.
   - Updates _depth = Math.Max(_depth, node.Depth + 1).

2. `void RemoveNode(string namespacePrefix)`:
   - Removes a routing entry. Used during shard decommission or federation merge.

3. `async Task<FederationResolveResult> ResolveAsync(string namespacePath, CancellationToken ct = default)`:
   - **Longest-prefix match**: Iterate _routingTable in reverse (longest first) to find the longest prefix that matches namespacePath.
   - If no match found, throw `InvalidOperationException($"No federation node handles namespace path '{namespacePath}'")`.
   - Recursively call `matchedNode.ResolveAsync(namespacePath, ct)`.
   - Return result with HopsUsed incremented by 1.
   - Enforce max hop limit of 10 (configurable). If exceeded, throw `InvalidOperationException("Federation resolution exceeded maximum hop count")` to prevent infinite recursion from misconfigured circular references.

4. `async Task<IReadOnlyList<IFederationNode>> GetChildrenAsync(CancellationToken ct = default)`:
   - Return _routingTable.Values as list.

5. `async Task<FederationNodeStats> GetStatsAsync(CancellationToken ct = default)`:
   - Fan out GetStatsAsync to all children via Task.WhenAll.
   - Aggregate: sum TotalShardCount, TotalObjectCount, TotalCapacityBytes, UsedCapacityBytes. MaxDepth = max of children's MaxDepth + 1.

6. `IFederationNode? FindNode(Guid nodeId)`:
   - Check _nodesByIdCache. If not found, recursively search children.
   - Returns null if not found.

Static factory for common topology:
```csharp
public static SuperFederationRouter CreateTwoLevel(
    Guid superFedId,
    IReadOnlyList<(string prefix, Guid fedId, VdeFederationRouter router)> federations)
```
- Creates a super-federation with FederationNode children, one per prefix.
- Convenience for the most common 2-level setup.

Design notes:
- Longest-prefix match uses SortedList iteration which is O(n) for n routing entries. At super-federation level, n is expected to be small (tens to hundreds of federations). If scale demands it, a trie can replace this later.
- MaxHops=10 prevents infinite recursion from circular federation references (which should not happen but must be defended against).
- Single-VDE deployments never instantiate SuperFederationRouter (it is only created when admin explicitly configures multi-federation).

Walks a federation node tree (IFederationNode) to produce an admin-visible snapshot of the entire federation topology.

Constructor: `FederationTopology(IFederationNode root)`

Methods:

1. `async Task<FederationTopologyNode> BuildAsync(CancellationToken ct = default)`:
   - Recursively walk the IFederationNode tree starting from root.
   - For each node: get stats, get children, recurse into children.
   - Build FederationTopologyNode tree.
   - Enforce max depth of 10 to prevent runaway recursion.
   - If root is a LeafVdeNode, return single node with empty Children.

2. `async Task<FederationTopologySummary> SummarizeAsync(CancellationToken ct = default)`:
   - Call BuildAsync, then compute summary stats:
   ```csharp
   [SdkCompatibility("6.0.0")]
   public readonly record struct FederationTopologySummary(
       int TotalNodes,
       int LeafCount,
       int FederationCount,
       int SuperFederationCount,
       int MaxDepth,
       long TotalShardCount,
       long TotalObjectCount,
       long TotalCapacityBytes,
       long UsedCapacityBytes,
       double UtilizationPercent);
   ```
   - Walk the built tree counting nodes by type, computing totals.
   - UtilizationPercent = TotalCapacityBytes > 0 ? (double)UsedCapacityBytes / TotalCapacityBytes * 100.0 : 0.0

3. `static string FormatAsTree(FederationTopologyNode root, int indent = 0)`:
   - Produces a human-readable tree string for CLI/admin display:
   ```
   [SuperFed] super-001 (depth=0, 3 children)
     [Fed] fed-east (depth=1, shards=500, 1.2 TB)
       [Leaf] shard-001 (depth=2, objects=50K, 120 GB)
       [Leaf] shard-002 (depth=2, objects=48K, 115 GB)
       ...
     [Fed] fed-west (depth=1, shards=300, 800 GB)
       ...
   ```
   - Use StringBuilder for efficiency.
   - Truncate child display at 10 items with "... and N more" for large federations.

4. `async Task<IReadOnlyList<FederationTopologyNode>> FindByTypeAsync(FederationNodeType nodeType, CancellationToken ct = default)`:
   - Build tree, then flatten and filter by node type.
   - Useful for admin queries like "show me all leaf shards" or "show me all federations".

Design: FederationTopology is read-only and produces snapshots. It does not modify the routing tree. All stats are point-in-time (may be slightly stale by the time admin reads them, which is acceptable for admin inspection).
