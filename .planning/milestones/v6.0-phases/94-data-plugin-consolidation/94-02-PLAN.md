<objective>
Consolidate all catalog storage to UltimateDataCatalog as the single authoritative asset catalog.

Purpose: UltimateDataLake stores catalog entries in a local `_catalog` BoundedDictionary (using SDK's `DataCatalogEntry` type) with add/get/list/remove operations -- duplicating what UltimateDataCatalog already provides authoritatively via `catalog.register`, `catalog.search`, `catalog.discover` message bus topics. Additionally, `FabricDataCatalogStrategy` in UltimateDataManagement has an inline metadata management implementation that should delegate to the authoritative catalog. This plan replaces both with bus delegation.

Output: Two modified files with bus delegation replacing local catalog storage.
</objective>

  <name>Task 1: Replace UltimateDataLake local catalog with bus delegation</name>
  <files>Plugins/DataWarehouse.Plugins.UltimateDataLake/UltimateDataLakePlugin.cs</files>
  <action>
Replace the `HandleCatalogAsync` method in UltimateDataLakePlugin (lines ~259-296) to delegate to UltimateDataCatalog via message bus instead of using the local `_catalog` BoundedDictionary.

Current implementation: Stores `DataCatalogEntry` objects in `_catalog` dictionary with add/get/list/remove actions locally.

New implementation:
1. Replace the `HandleCatalogAsync` method body. For each action:
   - "add": Create a PluginMessage with type "catalog.register" and payload {action="add", id, name, location, format, zone (from original payload fields)}. Send via `await MessageBus.SendAsync("catalog.register", msg)`. On success, copy response entry to `message.Payload["entry"]`. On failure, set error.
   - "get": Create a PluginMessage with type "catalog.search" and payload {action="get", id}. Send via `await MessageBus.SendAsync("catalog.search", msg)`. Copy response to `message.Payload["entry"]`.
   - "list": Create a PluginMessage with type "catalog.search" and payload {action="list"}. Send via `await MessageBus.SendAsync("catalog.search", msg)`. Copy response to `message.Payload["entries"]`.
   - "remove": Create a PluginMessage with type "catalog.register" and payload {action="remove", id}. Send via `await MessageBus.SendAsync("catalog.register", msg)`.

2. Wrap each SendAsync call in try/catch for graceful degradation. When MessageBus is null or call fails, return `message.Payload["success"] = false` with error description.

3. Make the method `async Task` (currently returns Task synchronously).

4. Remove the `_catalog` field declaration (line 40: `private readonly BoundedDictionary<string, DataCatalogEntry> _catalog`).

5. Remove all other references to `_catalog`:
   - In `GetMetadata()` (line ~213): remove `metadata["CatalogEntries"] = _catalog.Count;`
   - In `HandleStatsAsync` (line ~421): remove `message.Payload["catalogEntries"] = _catalog.Count;`
   - In state persistence/restore: remove `_catalog` serialization (line ~474-478) and snapshot (line ~523)
   - In `Dispose` (line ~556): remove `_catalog.Clear()`

6. Note: Plan 94-01 removes `_lineage` from this same file. If 94-01 runs first, `_lineage` will already be gone. If 94-02 runs first, leave `_lineage` for 94-01. Both plans are Wave 1 but touch different fields and methods, so there is minimal conflict. However, if both run simultaneously, the executor should handle merge carefully. The `HandleCatalogAsync` method (this task) and `HandleLineageAsync` method (94-01 Task 2) are distinct methods. The shared concern is only in Dispose, GetMetadata, state persistence, and field declarations. If 94-01 already ran, adapt -- do not re-add removed _lineage references.

7. Keep the catalog strategies in `Strategies/Catalog/DataCatalogStrategies.cs` (MetadataCatalogStrategy, DataDiscoveryStrategy, BusinessGlossaryStrategy, etc.) -- these are data-lake-specific catalog analysis strategies, NOT duplicate stores.
