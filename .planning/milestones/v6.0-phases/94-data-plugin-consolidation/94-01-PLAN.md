<objective>
Consolidate all lineage storage to UltimateDataLineage as the single authoritative lineage store.

Purpose: UltimateDataCatalog stores lineage in a local `_relationships` BoundedDictionary and UltimateDataLake stores lineage in a local `_lineage` BoundedDictionary. Both are duplicate, divergent implementations of what UltimateDataLineage already provides authoritatively via `lineage.track`, `lineage.upstream`, `lineage.downstream`, `lineage.add-node`, `lineage.add-edge` message bus topics. This plan replaces both local stores with async message bus delegation to UltimateDataLineage.

Output: Two modified plugin files with bus delegation replacing local lineage storage.
</objective>

  <name>Task 1: Replace UltimateDataCatalog local lineage with bus delegation</name>
  <files>Plugins/DataWarehouse.Plugins.UltimateDataCatalog/UltimateDataCatalogPlugin.cs</files>
  <action>
Replace the `HandleLineageAsync` method in UltimateDataCatalogPlugin (lines ~317-351) to delegate to UltimateDataLineage via message bus instead of using the local `_relationships` BoundedDictionary.

Current implementation: Stores `CatalogRelationship` objects in `_relationships` dictionary, handles add/upstream/downstream/list actions locally.

New implementation:
1. Replace the `HandleLineageAsync` method body. For each action:
   - "add": Create a PluginMessage with type "lineage.add-edge" and payload fields {sourceNodeId=sourceId, targetNodeId=targetId, edgeType=type}. Send via `MessageBus.SendAsync("lineage.add-edge", msg)`. If MessageBus is null or response fails, set `message.Payload["success"] = false` and `message.Payload["error"] = "Lineage service unavailable"`.
   - "upstream": Create a PluginMessage with type "lineage.upstream" and payload {nodeId=targetId}. Send via `MessageBus.SendAsync("lineage.upstream", msg)`. Copy response payload to message.Payload["relationships"].
   - "downstream": Create a PluginMessage with type "lineage.downstream" and payload {nodeId=sourceId}. Send via `MessageBus.SendAsync("lineage.downstream", msg)`.  Copy response payload to message.Payload["relationships"].
   - "list": Create a PluginMessage with type "lineage.search" and payload {query="*"}. Send via `MessageBus.SendAsync("lineage.search", msg)`. Copy response to message.Payload["relationships"].

2. Wrap each SendAsync call in try/catch. On exception (including when MessageBus is null), return `message.Payload["success"] = false` with error description. This is the graceful degradation pattern.

3. Make the method `async Task` (currently returns Task synchronously).

4. Remove the `_relationships` field declaration (line ~37: `private readonly BoundedDictionary<string, CatalogRelationship> _relationships`).

5. Remove all other references to `_relationships`:
   - In `GetMetadata()` (line ~214): remove `metadata["CatalogRelationships"] = _relationships.Count;`
   - In `HandleStatsAsync` (line ~469): remove `message.Payload["catalogRelationships"] = _relationships.Count;`
   - In state persistence (line ~536): remove `_relationships` serialization/deserialization
   - In `OnBeforeStatePersistAsync` (line ~563): remove `_relationships` snapshot
   - In `Dispose` (line ~580): remove `_relationships.Clear()`

6. The `CatalogRelationship` record type (lines ~606-614) can remain since it may be used by the LivingCatalog strategies internally (those are separate from the plugin-level store).

7. Do NOT touch the `CatalogScalingManager` `_relationships` cache -- that is a scaling concern separate from the plugin-level duplicate store. It will be addressed if needed in plan 94-05.

Note: The `SendAsync` pattern uses `MessageResponse` which has `Success`, `Payload`, `ErrorMessage`. Check `response.Success` before extracting data.
