The engine is a sealed class with constructor parameters:
- `IReadOnlyList<PlacementPolicy> policies` -- ordered by tier (hot first, frozen last); engine selects best match
- `Func<CancellationToken, ValueTask<IReadOnlyList<CompoundBlockDeviceInfo>>> deviceDiscovery` -- discovers available storage nodes
Also define a `readonly record struct CompoundBlockDeviceInfo` (in same file) with: `Guid DeviceId`, `StorageTier Tier`, `string RackId`, `long CapacityBlocks`, `long UsedBlocks`, `bool IsHealthy`.

**Core methods:**

`ValueTask<ShardPlacement> ComputePlacementAsync(DataShardDescriptor shard, StorageTier requestedTier, CancellationToken ct = default)`:
1. Look up the PlacementPolicy matching `requestedTier`. If not found, throw `InvalidOperationException`.
2. Call `policy.Validate()`.
3. Call `deviceDiscovery` to get available devices.
4. Filter devices to those matching `requestedTier` AND `IsHealthy`.
5. If `policy.RackAware`, apply rack-diversity: greedily select devices from distinct racks. Use a `Dictionary<string, List<CompoundBlockDeviceInfo>>` grouped by RackId. Round-robin across racks.
6. For replica-based (Hot/Warm): select `policy.ReplicaCount` devices. First device = Primary role, rest = Replica role.
7. For erasure-coded (Cold/Frozen): select `policy.ErasureDataShards + policy.ErasureParityShards` devices. First N = ErasureData role, last M = ErasureParity role.
8. If not enough devices after rack-aware filtering, fall back to best-effort (allow same-rack) but log a warning via a `PlacementWarning` field on ShardPlacement.
9. Calculate `EstimatedMonthlyCostUsd` = sum of (device capacity share * tier cost per GB).
10. If cost > `policy.MaxCostPerGbMonth * shard.CapacityBlocks * blockSizeBytes / 1GB`, throw `InvalidOperationException("Placement exceeds cost ceiling")`. Use a `blockSizeEstimateBytes` parameter (default 4096).
11. Return `ShardPlacement` with all targets, tier, policy, cost, and UTC ticks.

`StorageTier RecommendTier(DataShardDescriptor shard, ShardAccessPattern accessPattern)`:
- Define `readonly record struct ShardAccessPattern` (in same file) with: `double ReadsPerSecond`, `double WritesPerSecond`, `double AvgLatencyMs`, `long LastAccessUtcTicks`.
- Logic: if accessPattern.ReadsPerSecond > 1000 OR WritesPerSecond > 100 -> Hot. If ReadsPerSecond > 10 -> Warm. If LastAccessUtcTicks older than 30 days -> Frozen. Else Cold. This is the simple default; users override via explicit policy.

**Single-VDE optimization:** Add a static method `ShardPlacement CreatePassthrough(Guid shardVdeId, Guid deviceId)` that returns a minimal ShardPlacement with Hot tier, 1 target (Primary), zero cost. This is used when FederationMode is SingleVde -- no engine evaluation needed.

Add `PlacementWarning` as a `string?` property on ShardPlacement (update the record struct definition from Task 1 -- add `string? Warning` field, default null). Use `with` expression if needed.

Thread safety: the engine is stateless per call (all state comes from parameters and deviceDiscovery). No locking needed.
