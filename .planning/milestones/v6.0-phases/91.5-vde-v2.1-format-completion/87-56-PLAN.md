Create `DirectFileBlockDevice` sealed class with `[SdkCompatibility("6.0.0")]` implementing `IDirectBlockDevice`:
- Constructor: `string path, int blockSize, long blockCount, bool createNew, int alignmentRequirement = 4096`
- Opens file handle with platform-specific flags:
  - On all platforms: use `File.OpenHandle` with `FileOptions.Asynchronous | FileOptions.RandomAccess | FileOptions.WriteThrough`
  - Add `[SupportedOSPlatform("windows")]` path: add `FileOptions.None` -- the NO_BUFFERING flag is applied via the `preallocationSize` overload trick is not possible; instead use `const FileOptions NoBuffering = (FileOptions)0x20000000` (FILE_FLAG_NO_BUFFERING raw value) and bitwise OR it into options. Document this cast clearly.
  - On Linux/macOS: O_DIRECT cannot be set via File.OpenHandle directly. Use a P/Invoke approach: open the file normally, then use `fcntl` to add O_DIRECT (Linux) or `F_NOCACHE` (macOS). Or, simpler approach: just use `FileOptions.WriteThrough` which does `FILE_FLAG_WRITE_THROUGH` on Windows (close to NO_BUFFERING for writes) and document that full O_DIRECT on Linux requires the IoUringBlockDevice or a future P/Invoke enhancement. The `IsDirectIo` property returns `true` on Windows (where NO_BUFFERING is applied) and `false` on Linux/macOS (where this is a best-effort bypass via WriteThrough). This is acceptable because IoUringBlockDevice handles true O_DIRECT on Linux and KqueueBlockDevice handles F_NOCACHE on macOS.
- `BlockSize`, `BlockCount`, `IsDirectIo`, `AlignmentRequirement`, `MaxBatchSize = 64`
- `ReadBlockAsync` / `WriteBlockAsync` / `FlushAsync`: delegate to `RandomAccess` API (same pattern as `FileBlockDevice` but with alignment enforcement -- verify buffer alignment in debug builds via `Debug.Assert`)
- `ReadBatchAsync` / `WriteBatchAsync`: loop over requests sequentially using `RandomAccess.ReadAsync`/`WriteAsync`, returning count of completed operations. On failure of any operation, return count completed so far (partial success).
- `GetAlignedBuffer(int byteCount)`: returns `new AlignedMemoryOwner(byteCount, AlignmentRequirement)`
- `DisposeAsync`: dispose handle and semaphore (same pattern as FileBlockDevice)

Create `BlockDeviceFactory` static class with `[SdkCompatibility("6.0.0")]`:
- `static IBlockDevice Create(string path, int blockSize, long blockCount, bool createNew, BlockDeviceOptions? options = null)`
- 7-step auto-detection cascade (per VDE spec AD-46):
  1. If `ForceImplementation` is set, use that directly
  2. If SPDK available (check not implemented yet -- always false for now) -> log and skip
  3. If Linux AND io_uring available -> return IoUringBlockDevice (check `RuntimeInformation.IsOSPlatform(OSPlatform.Linux)`, wrap in try/catch for unavailability)

  5. If Windows 10+ -> skip WindowsOverlapped (not yet implemented, log at Debug)
  6. If macOS/FreeBSD AND kqueue -> skip (not yet implemented, log at Debug)
  7. If `options.DirectIo` -> return `DirectFileBlockDevice`
  8. Else -> return `FileBlockDevice` (baseline, always available)
- Use `Microsoft.Extensions.Logging.ILogger?` parameter (nullable, optional) for logging which implementation was selected at Debug level
- The factory should be straightforward -- just a cascading if/else. Steps 2-6 initially return the next step since those implementations come in later plans (87-57, 87-58, 87-59, 87-60). Add `// TODO: 87-57` style comments for each placeholder.
