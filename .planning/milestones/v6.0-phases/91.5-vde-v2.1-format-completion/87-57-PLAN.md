All classes in namespace `DataWarehouse.SDK.VirtualDiskEngine.IO.Windows`. All classes get `[SupportedOSPlatform("windows")]` and `[SdkCompatibility("6.0.0")]`.

**OverlappedNativeMethods** internal static class:
- P/Invoke to kernel32.dll:
  - `CreateFileW` (returns SafeFileHandle)
  - `CreateIoCompletionPort` (returns IntPtr IOCP handle)
  - `GetQueuedCompletionStatusEx` (batch dequeue from IOCP)
  - `ReadFile` with OVERLAPPED* parameter
  - `WriteFile` with OVERLAPPED* parameter
  - `FlushFileBuffers`
  - `CloseHandle`
- Constants: `FILE_FLAG_OVERLAPPED = 0x40000000`, `FILE_FLAG_NO_BUFFERING = 0x20000000`, `FILE_FLAG_WRITE_THROUGH = 0x80000000`, `GENERIC_READ = 0x80000000`, `GENERIC_WRITE = 0x40000000`, `OPEN_EXISTING = 3`, `CREATE_NEW = 1`
- `OVERLAPPED_ENTRY` struct for `GetQueuedCompletionStatusEx`

**WindowsOverlappedBlockDevice** sealed class implementing `IBatchBlockDevice`:
- Constructor: `string path, int blockSize, long blockCount, bool createNew, int concurrency = 64`
- Opens file with `CreateFileW` using `FILE_FLAG_OVERLAPPED | FILE_FLAG_NO_BUFFERING | FILE_FLAG_WRITE_THROUGH`
- Creates IOCP via `CreateIoCompletionPort`, associates file handle
- `BlockSize`, `BlockCount`, `MaxBatchSize = concurrency`
- `ReadBlockAsync`: allocates `NativeOverlapped` via `ThreadPool.UnsafeRegisterNativeIOCompletionCallback` pattern. Since .NET provides managed IOCP integration via `SafeFileHandle` + `RandomAccess`, use a simpler approach: open with `File.OpenHandle(path, mode, access, FileShare.None, FileOptions.Asynchronous | (FileOptions)0x20000000 | FileOptions.WriteThrough)` and use `RandomAccess.ReadAsync/WriteAsync` which internally uses OVERLAPPED. This gives IOCP performance without raw P/Invoke complexity. The key difference from `DirectFileBlockDevice` is that this class is explicitly Windows-optimized and adds batch coalescing.
- `WriteBatchAsync`: submit all write operations via `Task.WhenAll` on `RandomAccess.WriteAsync` calls for parallelism, return count of successful completions
- `ReadBatchAsync`: submit all read operations via `Task.WhenAll` on `RandomAccess.ReadAsync` calls for parallelism, return count of successful completions
- `FlushAsync`: `RandomAccess.FlushToDisk`
- `DisposeAsync`: dispose handle, cleanup resources

**IoRingNativeMethods** internal static class:
- P/Invoke to kernel32.dll (Windows 11+):
  - `CreateIoRing` -> creates HIORING handle
  - `CloseIoRing`
  - `BuildIoRingReadFile` -> enqueues read SQE
  - `BuildIoRingWriteFile` -> enqueues write SQE  (not available in all builds -- check availability)
  - `SubmitIoRing` -> submits queued SQEs, waits for completions
  - `BuildIoRingRegisterFileHandles` -> register file handles for fixed-handle I/O
  - `BuildIoRingRegisterBuffers` -> register buffers for zero-copy
- Structs: `IORING_CREATE_FLAGS`, `IORING_BUFFER_INFO`, `IORING_CQE`
- `static bool IsSupported()` -- attempts to call `CreateIoRing` and closes immediately. Cache result in a `Lazy<bool>`. Returns false if DllNotFoundException or EntryPointNotFoundException.

**IoRingBlockDevice** sealed class implementing `IBatchBlockDevice`:
- Constructor: `SafeFileHandle fileHandle, int blockSize, long blockCount, int ringSize = 64`
- Creates IoRing via `CreateIoRing` with `ringSize` entries
- Registers file handle and optionally pre-allocated buffers
- `MaxBatchSize = ringSize`
- `ReadBlockAsync`: build single SQE via `BuildIoRingReadFile`, submit via `SubmitIoRing` with waitCount=1, copy from registered buffer to caller buffer
- `WriteBlockAsync`: build SQE via `BuildIoRingWriteFile` (if available, else fall back to `RandomAccess.WriteAsync`), submit
- `ReadBatchAsync`: build multiple SQEs, submit with waitCount=N, process CQEs
- `WriteBatchAsync`: same pattern for writes
- `FlushAsync`: flush via file handle
- `DisposeAsync`: close IoRing handle, free registered buffers
- Static `IsSupported()` delegates to `IoRingNativeMethods.IsSupported()`
- If IoRing creation fails at runtime, throw `PlatformNotSupportedException` with message suggesting WindowsOverlappedBlockDevice

NOTE: IoRing write API (`BuildIoRingWriteFile`) was added in later Windows 11 builds. If only read is available, implement writes via `RandomAccess.WriteAsync` fallback and document this limitation.
