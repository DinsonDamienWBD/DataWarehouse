**SpdkDmaAllocator** sealed class:
- `IMemoryOwner<byte> Allocate(int byteCount, int alignment = 4096)`:
  - Calls `spdk_dma_malloc(byteCount, alignment, IntPtr.Zero)`
  - Returns a `SpdkDmaMemoryOwner` (private nested class implementing `IMemoryOwner<byte>`)
  - `SpdkDmaMemoryOwner.Memory` wraps the DMA pointer via an `UnmanagedMemoryManager<byte>` (same pattern as AlignedMemoryOwner)
  - `SpdkDmaMemoryOwner.Dispose()` calls `spdk_dma_free`
- `void Free(IntPtr buffer)` -- direct free for callers managing their own pointers
- Thread-safe, no internal pooling (SPDK manages its own hugepage pool)

**SpdkBlockDevice** sealed class implementing `IDirectBlockDevice`:
- Constructor: `string pciAddress, uint namespaceId = 1, int queueDepth = 256`
  - `pciAddress` is the PCIe BDF (e.g., "0000:01:00.0") of the NVMe device bound to vfio-pci
  - Init sequence:
    1. `spdk_env_opts_init` + `spdk_env_init` (only once, use static `Lazy<bool>` guard)
    2. `spdk_nvme_probe` with transport ID set to `pciAddress` -- attach callback stores controller handle
    3. `spdk_nvme_ctrlr_get_ns(ctrlr, namespaceId)` -- get namespace
    4. Query sector size and sector count from namespace
    5. Allocate I/O queue pair via `spdk_nvme_ctrlr_alloc_io_qpair`
  - Sets `BlockSize` from `spdk_nvme_ns_get_sector_size`, `BlockCount` from `spdk_nvme_ns_get_num_sectors`
- Properties:
  - `IsDirectIo = true`
  - `AlignmentRequirement = BlockSize` (NVMe requires sector-aligned DMA)
  - `MaxBatchSize = queueDepth`
- `GetAlignedBuffer(int byteCount)`: uses `SpdkDmaAllocator.Allocate(byteCount, AlignmentRequirement)`
- `ReadBlockAsync`:
  1. Allocate DMA buffer via `SpdkDmaAllocator` (or accept pre-allocated if caller used `GetAlignedBuffer`)
  2. Submit `spdk_nvme_ns_cmd_read` with completion callback
  3. Poll `spdk_nvme_qpair_process_completions` in a loop with `Task.Yield()` between iterations
  4. On completion callback: set `TaskCompletionSource` result
  5. Copy from DMA buffer to caller's buffer if not same pointer
  6. Free DMA buffer if locally allocated
- `WriteBlockAsync`: same pattern with `spdk_nvme_ns_cmd_write`
- `ReadBatchAsync`: submit multiple read commands to the queue pair, poll for all completions. SPDK supports inflight commands up to queue depth.
- `WriteBatchAsync`: same for writes
- `FlushAsync`: submit NVMe flush command (or no-op if SPDK handles durability)
- `DisposeAsync`: free queue pair, detach controller, free DMA allocator resources
- Static `IsSupported()` delegates to `SpdkNativeMethods.IsSupported()`

IMPORTANT: The completion callback from SPDK runs on the SPDK polling thread (the thread calling `process_completions`). Use `TaskCompletionSource` with `TaskCreationOptions.RunContinuationsAsynchronously` to avoid blocking the SPDK thread. The polling loop should be `while (!tcs.Task.IsCompleted) { spdk_nvme_qpair_process_completions(qpair, 0); await Task.Yield(); }`.

NOTE: SPDK is primarily a Linux bare-metal technology. The code will compile on all platforms (P/Invoke is just declarations) but will only function on systems with libspdk installed and NVMe devices bound to vfio-pci. `IsSupported()` handles runtime detection.
