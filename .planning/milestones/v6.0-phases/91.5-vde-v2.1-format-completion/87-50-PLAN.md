Methods:
- `async Task<PreambleVerificationResult> VerifyAsync(Stream stream, CancellationToken ct)`:
  1. Read 64-byte header from position 0
  2. Validate magic bytes == "DWVD-BOO" → if not, return NotAPreamble
  3. Compute BLAKE3(bytes[0..55]), truncate to 8 bytes, compare to HeaderChecksum field at bytes 56-63
     → if mismatch, return HeaderCorrupted
  4. Read PreambleVersion — if > 1, return UnsupportedVersion (future-proofing)
  5. Read the 32-byte content hash at ContentHashOffset
  6. Stream through kernel+SPDK+runtime payload bytes (KernelOffset..KernelOffset+KernelSize, SpdkOffset..SpdkOffset+SpdkSize, RuntimeOffset..RuntimeOffset+RuntimeSize), compute BLAKE3 incrementally
  7. Compare computed hash to stored hash → if mismatch, return ContentCorrupted
  8. If all pass → return Valid

- `readonly struct PreambleVerificationResult`:
  - `PreambleVerificationStatus Status` enum: Valid, NotAPreamble, HeaderCorrupted, ContentCorrupted, UnsupportedVersion, IoError
  - `string? ErrorDetail` — human-readable description of failure
  - `PreambleHeader? Header` — parsed header if available (null when NotAPreamble)
  - `byte[]? ExpectedContentHash` — stored hash (for diagnostics)
  - `byte[]? ActualContentHash` — computed hash (for diagnostics)
  - `bool IsValid => Status == PreambleVerificationStatus.Valid`

- `async Task<bool> QuickVerifyHeaderOnlyAsync(Stream stream, CancellationToken ct)`:
  Fast path: only checks header checksum (no content hash). Used during VDE open — full content verification is expensive and only needed before boot or explicit `dw preamble verify`.

- Use BLAKE3 via the same approach as PreambleReader (System.Security.Cryptography SHA256 with BLAKE3 TODO if no BLAKE3 NuGet available, or use BLAKE3 if the SDK already references it). Check the SDK's .csproj for existing hash library references and use whatever is available.

Primary method:
- `async Task<PreambleUpdateResult> UpdateAsync(Stream vdeStream, PreambleUpdateOptions options, CancellationToken ct)`:
  1. Read existing preamble header from position 0
  2. Verify header integrity (quick verify) — if corrupt and !options.Force, return error
  3. Determine what's changing: kernel, SPDK, runtime, or all
  4. If new payload sizes differ from existing:
     a. Compute new layout via PreambleCompositionEngine.ComputeLayout
     b. If new VdeOffset > existing VdeOffset: REJECT — would require moving all VDE blocks (not supported in update, only at creation)
     c. If new VdeOffset <= existing VdeOffset: preamble fits in existing space
     d. If new VdeOffset < existing VdeOffset: zero-fill gap between new preamble end and VDE start
  5. Write new payload(s) at computed offsets
  6. Compute new BLAKE3 content hash over all payloads, write at ContentHashOffset
  7. Build new PreambleHeader with updated sizes/offsets
  8. Write header (PreambleWriter computes HeaderChecksum)
  9. Flush stream
  10. Return success with before/after sizes

- `sealed class PreambleUpdateOptions`:
  - `byte[]? NewKernelImage { get; init; }` — null = keep existing
  - `byte[]? NewSpdkBundle { get; init; }` — null = keep existing
  - `byte[]? NewRuntimeBinary { get; init; }` — null = keep existing
  - `bool Force { get; init; } = false` — update even if existing preamble is corrupt
  - `PreambleCompositionProfile? Profile { get; init; }` — optional profile override for flags
  - `bool ValidateAfterUpdate { get; init; } = true` — run full verification after update

- `readonly struct PreambleUpdateResult`:
  - `bool Success`
  - `string? ErrorMessage`
  - `ulong PreviousPreambleSize, ulong NewPreambleSize`
  - `PreambleVerificationResult? PostUpdateVerification` — if ValidateAfterUpdate was true

- `async Task<PreambleUpdateResult> UpdateFromFilesAsync(Stream vdeStream, string? kernelPath, string? spdkPath, string? runtimePath, bool force, CancellationToken ct)`:
  Convenience method: reads files from disk, delegates to UpdateAsync. This is the method the CLI `dw preamble update` command calls.

Crash safety (per spec):
- The header is written LAST, after all payload bytes and content hash are flushed
- If crash occurs before header write: old header is intact, old content hash matches old payloads → VDE opens normally with old preamble
- If crash occurs during header write: header checksum will fail → preamble skipped, VDE opens in hosted mode
- `dw preamble update --force` can recover from corrupted mid-update state
