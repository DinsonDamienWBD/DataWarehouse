**KqueueBlockDevice** sealed class implementing `IBatchBlockDevice`:

- Constructor: `string path, int blockSize, long blockCount, bool createNew, int alignmentRequirement = 4096`

Core I/O pattern:
- `ReadBlockAsync`: allocate `aiocb`, set `aio_fildes`, `aio_offset = blockNumber * BlockSize`, `aio_buf` pointing to caller's buffer (pinned via `GCHandle`), `aio_nbytes = BlockSize`. Set `aio_sigevent.sigev_notify = SIGEV_NONE`. Submit via `aio_read`. Poll completion via `aio_error` in a loop with `Task.Yield()` between iterations (or use a background polling thread). On completion, `aio_return` to get bytes read. Release `GCHandle`.
- `WriteBlockAsync`: same pattern with `aio_write` and caller's data buffer
- `FlushAsync`: call `fsync(fd)`

Batch I/O:
- `ReadBatchAsync`: submit all `aio_read` operations, register EVFILT_AIO events on the kqueue for each aiocb, call `kevent` to wait for completions in batch. Return count of successful operations.
- `WriteBatchAsync`: same pattern with `aio_write`

For the kevent-based batch completion:
1. Submit all AIO operations (aio_read/aio_write for each request)
2. Register each aiocb as a kevent change with EVFILT_AIO filter
3. Call `kevent()` with the changelist and eventlist to wait for completions
4. Process returned events, call `aio_return` for each completed operation
5. Return count of successes

Alternative simpler approach if kqueue+AIO integration is too complex: use `Task.WhenAll` over individual `ReadBlockAsync` calls (each using aio_read + polling). This is less efficient but correct. Implement the simpler approach first with a `// TODO: batch via kevent EVFILT_AIO` comment for future optimization.

- `DisposeAsync`: close kqueue fd, close file fd, free any pinned resources
- Static `IsSupported()` delegates to `KqueueNativeMethods.IsSupported()`

IMPORTANT: All native struct layouts must account for platform differences between macOS and FreeBSD (struct sizes may differ). Use `#if` or runtime OS detection where struct layouts diverge.
